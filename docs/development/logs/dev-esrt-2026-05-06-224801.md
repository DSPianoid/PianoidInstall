#### What "per-band tail smoothing" would mean architecturally

To make tail smoothing truly per-band, the fade-out logic must move INTO `band_processing.process_band` (added as a new step at the END of the per-band pipeline, after `skip_start_ms`). The scenario averager's project-wide fade can either:
- **Option A** stay as-is (project-wide pre-fade still applied at average-time), and the per-band fade is layered on top inside `process_band`. Default behaviour preserved if per-band fade is `None`.
- **Option B** move OUT of the scenario averager entirely and live only inside `process_band`. The `averaged_responses/` files would no longer carry a fade. This is a bigger change with backward-compat implications (existing projects' `averaged_responses/` already have fade baked in; would need re-averaging).

**Option A is simpler and backwards-compatible.** Recommend Option A.

### Step 4 — start_fade_ms investigation — 20:08

Hypothesis: `skip_start_ms > 0` introduces a hard cut → step discontinuity from implicit zero before sample 0 to actual signal value at the new start → spectral leakage degrading ESPRIT.

Measurement procedure: ran `process_band` (with `apply_preemphasis=True`, `end_fade_ms=None` to isolate the start cut) on real PlyWoodTake1_1 averaged_responses ch1 across 4 scenarios × 4 bands that use non-zero skip (Ultra-Low/Low/Low-Mid/Mid). For each (scenario, band): measured `|signal[0]|` (the would-be discontinuity) vs `max|signal|` (peak) and reported the ratio.

Result table (per-band MAX jump-ratio across the 4 scenarios):

| Band | skip_ms | Max |x[0]| / peak | Worst scenario | Verdict |
|------|---------|-------------------|----------------|---------|
| Ultra-Low | 50 | **94.05%** | Scenario20 | STRONGLY recommended |
| Low | 30 | **95.03%** | Scenario0 | STRONGLY recommended |
| Low-Mid | 15 | **40.00%** | Scenario10 | STRONGLY recommended |
| Mid | 5 | **76.08%** | Scenario20 | STRONGLY recommended |

The discontinuity is **NOT marginal** — it routinely starts at 30–95% of the band peak. A step of that magnitude generates a `1/f` spectral skirt that fully overlaps the ESPRIT analysis band. Per-band start-fade is overwhelmingly justified.

**Recommendation: ADD `start_fade_ms` per-band field, mirror `end_fade_ms` UX exactly.**

Proposed defaults (proportional to `skip_start_ms` since both target the same start-pollution region):
- Ultra-Low: 10 ms (= 20 % of 50 ms skip — typical fade-in/skip ratio)
- Low: 10 ms (~33 % of 30 ms skip)
- Low-Mid: 5 ms (33 % of 15 ms)
- Mid: 2 ms (40 % of 5 ms — round up to capture the discontinuity)
- Mid-High through Top: None (no skip → no discontinuity → no fade-in needed)

Application order inside `process_band`: AFTER `skip_start_ms` (which establishes the new sample 0), BEFORE `end_fade_ms` (which acts on the suffix only — ordering is independent). The fade-in window is the FIRST half of a Hann: `np.hanning(2N)[:N]` so it ramps from 0 at sample 0 up to ~1 at sample N. This is the symmetric counterpart of the end-fade.

Symbol: `start_fade_ms` (column to the right of `skip_start_ms`, before `end_fade_ms`). UI: plain TextField + column-header tooltip — same pattern as the other two.

Algorithm:
```js
function backFillBandDefaults(band, defaultBand) {
  // For every nullable field that the backend default says has a value,
  // fall back to the default if the saved band lacks the key.
  // Conservative: only back-fill UNDEFINED keys; preserve explicit null
  // (user opted out by clearing the cell).
  const out = { ...band };
  for (const key of ["start_fade_ms", "end_fade_ms"]) {
    if (out[key] === undefined && defaultBand && defaultBand[key] !== undefined) {
      out[key] = defaultBand[key];
    }
  }
  return out;
}
```

Applied at the `syncFromBackend` line 515 (`bands: ec.bands || []`) — replace with:
```js
bands: (ec.bands || []).map((b) => {
```

### Step 1 — Investigation (docs-first)

**Where built-in presets live today:**
- `PianoidCore/pianoid_middleware/modal_adapter/esprit/band_processing.py` — `STANDARD_BANDS` and `EXTENDED_BANDS` Python literal lists of `FrequencyBand` dataclass instances.
- `PianoidCore/pianoid_middleware/modal_adapter/esprit_runner.py:44` — `BAND_PRESETS = {"standard_4band": STANDARD_BANDS, "extended_8band": EXTENDED_BANDS}` in-memory registry.
- `esprit_runner.py:736-755` `EspritRunner.get_band_presets()` — serialises the registry into JSON dicts (with all 12 keys per band including my Phase-1 `start_fade_ms`/`end_fade_ms`).
- `routes.py:1485` `GET /modal/band_presets` — returns the serialised dict.
- `useModalAdapter.js:664-679` — fetches `/modal/band_presets` once on mount, stores in `bandPresets` state + `bandPresetsRef`. Used by the EspritConfig dropdown AND by my `backFillBandsList` back-fill helper.
- `EspritConfig.jsx:259-272` `handlePresetChange` — when user selects a preset, looks up `bandPresets[presetName]` and pushes the bands array into config (calling `onConfigChange` → `stageEspritConfig` → flips `espritConfigDirty`). Selecting `"custom"` is a no-op for bands (preset label only).

**Where built-in presets are consumed at run time:**
- `_resolve_bands` (`esprit_runner.py:606-619`) — receives the params dict for an ESPRIT run. Checks `params.get("bands")` FIRST (explicit bands list) and only falls back to `BAND_PRESETS[preset_name]` if `bands` is absent. **The frontend always sends the explicit bands list when launching ESPRIT** (verified by tracing `runEsprit` → `axios.post('/modal/run_esprit', {..., bands: ...})`). So at run time, **the preset name is just a label**; the actual band data is pushed through verbatim. This is critical for the design: **user presets do NOT need to be registered in the Python `BAND_PRESETS` dict for run-time correctness** — they only need to flow through the listing endpoint so the dropdown can populate the bands array on selection.

**Where to store user presets — options weighed:**

| Option | Path | Cross-project? | Survives reinstall? | Cross-machine? | Reuses existing infrastructure? |
|--------|------|----------------|---------------------|----------------|--------------------------------|
| (a) Global home | `~/.pianoid/band_presets.json` | YES | YES | NO | NO — new user-config root |
| (b) Install-local | `<install>/.../user_presets.json` | YES | NO (lost on reinstall) | NO | YES |
| (c) Sibling-to-projects | `<projects_base>/.user_band_presets.json` (default `D:\modal_projects\.user_band_presets.json`) | YES | YES | YES (when projects_base is on a network/external drive — same conditions as the projects themselves) | YES — reuses `projects_base` env var + `set_projects_base` API |

`projects_base` (`modal_adapter.py:91`) is already an env-controlled (`PIANOID_PROJECTS_DIR`) configurable root with a runtime setter (`POST /modal/projects/set_base`). User presets sitting next to projects naturally follows the user's chosen data location. **My recommendation: (c).** It reuses existing infrastructure (no new home-dir root, no Windows/Linux home-dir surprises, no test-fixture-paths surprises). And it co-locates the user's "stuff" — projects + cross-project presets in one tree. The trade-off is that if the user moves `projects_base`, they take their presets with them; arguably a feature, not a bug.

**Validation rules for user-saved preset names** (proposed):
- Non-empty (after `.strip()`)
- Not in built-in reserved set: `{"standard_4band", "extended_8band", "custom"}`
- Length ≤ 64 chars (UI sanity)
- Filesystem-safe (avoid `/`, `\`, `:`, `<`, `>`, `|`, `?`, `*`, `\0` since the storage file embeds the name as a JSON key — JSON itself is safe but the dropdown rendering should not be tricked by control chars)
- Collision behaviour: by default a `name` collision OVERWRITES (POST is idempotent on the canonical name). Optionally accept `force: true` for explicitness; default behaviour SHOULD overwrite to be useful.

**Round-trip data shape:**
- User saves: client POSTs `{name: "MyBass", bands: [<full band dicts>]}`. Backend serialises `{"MyBass": [<bands>]}` into the JSON file.
- User loads list: GET `/modal/band_presets` returns merged `{...built-ins, "MyBass": [<bands>]}`. To distinguish, add an `is_user_saved` flag — but a flag on a dict's VALUE doesn't naturally serialise. Two options:
  - **Option 1**: extend the response shape from `{name: bands_list}` to `{name: {bands: [...], is_user_saved: bool}}`. **Breaks frontend** (existing `presetData.bands || presetData` pattern works for both shapes, but my back-fill `backFillBandsList(bands, presetName, presetMap)` expects `presetMap[name]` to be the bands list directly). I'd need to update the back-fill to accept either shape.
  - **Option 2**: add a sibling `_user_preset_names: ["MyBass"]` array at the top level of the response, so existing band-data plumbing stays unchanged. Frontend uses this list to identify user presets in the dropdown for italics + delete affordance.

  **Option 2 is non-breaking and simpler. Recommended.**

**Frontend dropdown UX options:**

| Option | Pattern | MUI-idiomatic? | Implementation cost |
|--------|---------|----------------|---------------------|
| (i) Italics + divider | Built-in presets first; `<Divider />`; "Custom"; `<Divider />`; user presets in italics | YES — MUI `Divider` is canonical inside `<Select>` | Low |
| (ii) Icon prefix | `<UserIcon />` next to user-preset entries | YES | Low |
| (iii) Grouped via ListSubheader | `<ListSubheader>Built-in</ListSubheader>` ... `<ListSubheader>Saved</ListSubheader>` | YES | Low |

**Recommendation: (i) italics + divider** — cleanest, no extra icon imports, clearly differentiates without visual noise. Matches MUI's standard "non-clickable separator" pattern.

**Delete affordance options:**

| Option | Pattern | UX cost |
|--------|---------|---------|
| (A) Inline `×` on hover | Each user-preset row in dropdown has a small delete icon revealed on hover | Low — but clicking `×` inside a Select's MenuItem is fiddly because the Select's onChange fires on item click; need to `e.stopPropagation()` |
| (B) "Manage presets…" dialog | Separate menu item "Manage saved presets…" opens a Dialog with list + delete buttons | Slightly more UI; cleaner separation; works for batch operations |
| (C) Right-click context menu | Right-click a user preset → "Delete" | Discoverable issues; mobile-unfriendly |

**Recommendation: (A) inline `×` on hover** — fastest user path; one-click delete with confirm dialog before commit. The `e.stopPropagation()` on the icon click is a known MUI pattern.

**New REST endpoints (proposed):**
- `POST /modal/band_presets/save` — body `{name: string, bands: [<band dicts>]}`. Returns `{message: "saved", name}`. Validates name + writes to user file. Idempotent (overwrites on name collision).
- `DELETE /modal/band_presets/<path:name>` — removes a single user preset by name. Returns `{message: "deleted", name}` or 404 if not found.
- `GET /modal/band_presets` (existing) extended: merged response with new top-level field `_user_preset_names: [...]`.

(No separate `GET /modal/band_presets/user` endpoint needed — the existing list is the single source.)
