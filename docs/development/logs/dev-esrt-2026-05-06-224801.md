#### What "per-band tail smoothing" would mean architecturally

To make tail smoothing truly per-band, the fade-out logic must move INTO `band_processing.process_band` (added as a new step at the END of the per-band pipeline, after `skip_start_ms`). The scenario averager's project-wide fade can either:
- **Option A** stay as-is (project-wide pre-fade still applied at average-time), and the per-band fade is layered on top inside `process_band`. Default behaviour preserved if per-band fade is `None`.
- **Option B** move OUT of the scenario averager entirely and live only inside `process_band`. The `averaged_responses/` files would no longer carry a fade. This is a bigger change with backward-compat implications (existing projects' `averaged_responses/` already have fade baked in; would need re-averaging).

**Option A is simpler and backwards-compatible.** Recommend Option A.

### Step 4 — start_fade_ms investigation — 20:08

Hypothesis: `skip_start_ms > 0` introduces a hard cut → step discontinuity from implicit zero before sample 0 to actual signal value at the new start → spectral leakage degrading ESPRIT.

Measurement procedure: ran `process_band` (with `apply_preemphasis=True`, `end_fade_ms=None` to isolate the start cut) on real PlyWoodTake1_1 averaged_responses ch1 across 4 scenarios × 4 bands that use non-zero skip (Ultra-Low/Low/Low-Mid/Mid). For each (scenario, band): measured `|signal[0]|` (the would-be discontinuity) vs `max|signal|` (peak) and reported the ratio.

Result table (per-band MAX jump-ratio across the 4 scenarios):

| Band | skip_ms | Max |x[0]| / peak | Worst scenario | Verdict |
|------|---------|-------------------|----------------|---------|
| Ultra-Low | 50 | **94.05%** | Scenario20 | STRONGLY recommended |
| Low | 30 | **95.03%** | Scenario0 | STRONGLY recommended |
| Low-Mid | 15 | **40.00%** | Scenario10 | STRONGLY recommended |
| Mid | 5 | **76.08%** | Scenario20 | STRONGLY recommended |

The discontinuity is **NOT marginal** — it routinely starts at 30–95% of the band peak. A step of that magnitude generates a `1/f` spectral skirt that fully overlaps the ESPRIT analysis band. Per-band start-fade is overwhelmingly justified.

**Recommendation: ADD `start_fade_ms` per-band field, mirror `end_fade_ms` UX exactly.**

Proposed defaults (proportional to `skip_start_ms` since both target the same start-pollution region):
- Ultra-Low: 10 ms (= 20 % of 50 ms skip — typical fade-in/skip ratio)
- Low: 10 ms (~33 % of 30 ms skip)
- Low-Mid: 5 ms (33 % of 15 ms)
- Mid: 2 ms (40 % of 5 ms — round up to capture the discontinuity)
- Mid-High through Top: None (no skip → no discontinuity → no fade-in needed)

Application order inside `process_band`: AFTER `skip_start_ms` (which establishes the new sample 0), BEFORE `end_fade_ms` (which acts on the suffix only — ordering is independent). The fade-in window is the FIRST half of a Hann: `np.hanning(2N)[:N]` so it ramps from 0 at sample 0 up to ~1 at sample N. This is the symmetric counterpart of the end-fade.

Symbol: `start_fade_ms` (column to the right of `skip_start_ms`, before `end_fade_ms`). UI: plain TextField + column-header tooltip — same pattern as the other two.

Algorithm:
```js
function backFillBandDefaults(band, defaultBand) {
  // For every nullable field that the backend default says has a value,
  // fall back to the default if the saved band lacks the key.
  // Conservative: only back-fill UNDEFINED keys; preserve explicit null
  // (user opted out by clearing the cell).
  const out = { ...band };
  for (const key of ["start_fade_ms", "end_fade_ms"]) {
    if (out[key] === undefined && defaultBand && defaultBand[key] !== undefined) {
      out[key] = defaultBand[key];
    }
  }
  return out;
}
```

Applied at the `syncFromBackend` line 515 (`bands: ec.bands || []`) — replace with:
```js
bands: (ec.bands || []).map((b) => {
```
