### ROOT CAUSE
`pianoid.reset()` resets the **transient runtime/playback state** — it silences currently-ringing
strings and clears the output buffer (`resetStringsState` zeroes q/q_prev per AddArraysWithCUDA.cpp:783-786;
`clearRecords` clears sound_record_index per :826-827). It does NOT touch the **synthesis parameter
values** (string coefficients, mode coefficients). So after the user clicks Reset, the strings keep
ringing-down then go quiet — but the moment a new note plays, the SAME preset parameters synthesize the
SAME sound. From the user's POV "the Reset button is not working": it does not zero the synthesis.

The user wants: "set all the strings to zeros and all the modes to zeros" = zero the PARAMETER matrices,
not the vibration state. The current handler addresses a different (and badly-named) concept.

### THE EDIT (drop in alongside the existing memsets)
```cpp
void Pianoid::resetStringsState() {
    resetFlag = true;
    // Zero the persistent 2nd-derivative state
    if (dev_sound_prev_diff) {
        cudaMemset(dev_sound_prev_diff, 0, 4 * sizeof(real));
    }
    // dev-reset FIX: zero the string DISPLACEMENT state directly (host-side), not only via the kernel
    // reset branch (MainKernel.cu:344 s_a=0 -> writeback). The kernel path only fires if the cooperative
    // addKernel actually cycles after reset; under live audio on a small-SM GPU the cooperative launch can
    // fail to cycle (cudaErrorCooperativeLaunchTooLarge, see dev-bug1rt f96e266), leaving displacement
    // non-zero -> "Reset doesn't zero it, sound continues." A direct memset makes Reset deterministic
    // regardless of cycle state, and covers ALL strings incl. output strings 128+ so the output tap is
    // silenced too.
    if (dev_string_state) {
        cudaMemset(dev_string_state, 0,
                   2 * init_params_.array_size * init_params_.num_string_arrays() * sizeof(real));
    }
    // NOTE: dev_limiter_peak is deliberately NOT zeroed here. [existing comment unchanged]
    // Zero mode running state (q, q_prev) directly — no preset swap needed
    resetModeRunningState();
}
```

### SIZE EXPRESSION — VERIFIED against the allocation (Pianoid.cu:410-414)
dev_string_state element count = `2 * init_params_.array_size * init_params_.num_strings /
init_params_.num_strings_in_array`. `num_string_arrays()` (Pianoid.cuh:61-62) = `num_strings /
num_strings_in_array` (div-by-zero-safe → 0). So bytes =
`2 * init_params_.array_size * init_params_.num_string_arrays() * sizeof(real)` matches EXACTLY. (Literal
form `2 * init_params_.array_size * init_params_.num_strings / init_params_.num_strings_in_array *
sizeof(real)` is equivalent.) Factor 2 = [current | prev] halves (kernel reads both, MainKernel.cu:347-348);
full-buffer zero clears both = complete displacement reset. ✓

### SYNC: NONE
The two existing memsets here (dev_sound_prev_diff, dev_mode_running) use plain `cudaMemset` with NO
cudaDeviceSynchronize. Match that fire-and-forget pattern (default stream, ordered before next kernel).
Do NOT add cudaDeviceSynchronize (would stall the audio thread).

### COVERAGE (covers H-R2)
The memset clears the ENTIRE buffer = ALL strings, piano (0-127) AND output (128+). So even if dev-d52b's
output-mask path was leaving output strings ringing after reset (H-R2), a full-buffer zero silences them.
Correct whether root cause is H-R1 (kernel not cycling) or H-R2.

Do NOT remove resetFlag=true or the kernel `if(*status==500){s_a=0;}` branch — they still correctly reset
the in-flight shared copy for the cycle that DOES run. The memset is additive insurance.
