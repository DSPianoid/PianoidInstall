### Step 1 findings — 2026-05-27T09:50:00Z
- **Threading verified**: Flask request thread runs `pianoid = initialize(...)` -> `Pianoid::Pianoid()` constructor creates `ASIOAudioDriver` (NO ASIO calls yet, just object allocation). Then Flask request handler spawns `threading.Thread(target=long_running_procedure, ...)`. Worker thread eventually calls `pianoid_->startAudioDriver()` (Pianoid.cu:727-751) -> `audioDriver->init()` -> `AsioAudioOutput::Init()` (AsioAudioInterface.cpp:578) -> `InitAsioDriver()` (522) -> `loadAsioDriver()` + `ASIOInit()` + `ASIOStart()` from the worker thread.
- **NO `CoInitialize*` calls anywhere in `AsioAudioInterface.cpp` or `ASIOAudioDriver.cpp`**. ASIO is COM-based; per Steinberg SDK ASIO drivers require apartment-threaded COM (STA) for the calling thread that invokes ASIOInit/ASIOStart and that hosts the buffer-switch callback.
- **utilities.py is NOT the home of `long_running_procedure`** — that helper lives at `backendServer.py:635`. utilities.py is a tiny matplotlib/folder helper module.
- **Launcher does NOT pipe backend stdout to disk** — it pipes to WebSocket broadcast only (launcher.js:228-234, `stdio: ['ignore', 'pipe', 'pipe']`). So when the backend dies during ASIO init, the stdout printf trail is lost unless the React console pane is open and someone scrolls back.
- **ASIO logging in ASIOAudioDriver.cpp + AsioAudioInterface.cpp uses raw `printf`, NOT PLOG**. Errors go to stdout, never to `logs/pianoid_init.log`. SDL3AudioDriver.cpp uses `PLOG`.
- **Live evidence from `PianoidCore/logs/pianoid_init.log`**: two recent ASIO sessions both reach `CycleTimeEstimator started: 48000 Hz, 64 samples/cycle, 1333 us/cycle` (which only logs from inside `OnlinePlaybackEngine::run`'s timing-start block AFTER `pianoid_->startAudioDriver()` returned). Second session died after 2.87s (`pianoid object being destroyed`). So `startAudioDriver` succeeded, `ASIOStart` succeeded, the run loop entered, callbacks were firing — and then something in the steady-state callback path crashed.
- **Cache file exists**: `pianoid_middleware/asio_driver_cache.txt` contains "UMC ASIO Driver". InitAsioDriverByName will be called with this name from the worker thread.

### Step 1 hypothesis update
- Original orchestrator hypothesis: STA/MTA COM apartment mismatch causing `ASIOInit/ASIOStart` failure. **Refined by evidence**: `ASIOStart` actually succeeded (cycle estimator logs prove startAudioDriver returned). The crash is later, during callback/cycle execution.
- New leading hypothesis: ASIO callback fires on the driver's own thread (the buffer-switch thread is owned by the ASIO driver itself, not by us). The buffer-switch callback runs `bufferSwitchTimeInfo` which calls `directOutputFn` (the consume from circular buffer). But our worker thread (the `long_running_procedure` thread) is meanwhile running the cycle loop and producing to the circular buffer. If the ASIO driver's bufferSwitch thread requires COM-initialised state to call back INTO `directOutputFn` (which itself reads ASIO globals via `asioDriverInfo`) — or, more likely, the worker thread's lack of COM apartment causes ASIO host->client message dispatching to fail, kAsioResetRequest fires, the engine tears down.
- ALTERNATIVELY: the issue is not COM at all but a producer/consumer race in the LockFree CircularBuffer (8 chunks) — but the diag report says standalone Python tests hit 2153 cycles cleanly, which suggests the circular buffer is fine when COM is properly set up in the calling thread.
- Will capture the actual stdout to discriminate between these in Step 0.

### Step 1 (continued): Discriminating diagnostic via `docs/development/diagnostics/dev-asiocrash-b20f-comtest.py`

**Standalone reproduction in fresh Python process (no Flask):**

| Test | Pattern | Result |
|---|---|---|
| 1 | Open + Close + Open + Close, all on MAIN thread, no COM init | BOTH OPEN succeed |
| 5 | Open+Close on Worker W1, then Open on Worker W2 (no COM init) | W1 OK, **W2 FAILS** (same crash as Flask) |
| 6 (MTA) | Open W1 + Close + Open W2 — both threads init COM as MTA | **BOTH FAIL** (MTA breaks ASIO) |
| 6 (STA) | Open W1 + Close + Open W2 — both threads init COM as STA (APARTMENTTHREADED) | **BOTH SUCCEED** |
| 7 | Open W1 STA + Close from MAIN (no COM) + Open W2 STA | **BOTH SUCCEED** (close from non-COM thread is fine) |

**ROOT CAUSE confirmed:** ASIO drivers (specifically Behringer UMC + tested with the enumeration sweep that hit Ableton Push, ESI GIGAPORT, Realtek too) require any thread that calls `ASIOInit/ASIOStart/ASIODisposeBuffers/ASIOExit` to have COM initialized as **STA (COINIT_APARTMENTTHREADED)**. When the engine's `PianoidUnifiedPlaybackThread` (the `run_online` closure at pianoid.py:1455-1473) opens ASIO without STA, the driver "appears" to work for that thread's lifetime, but when the thread exits its (default-uninitialized) COM apartment is torn down. This destroys the ASIO driver's COM-resident state. The NEXT worker thread that tries to open ASIO finds the driver unable to bind via `asioOpenDriver(...)` — so `loadAsioDriver()` returns FALSE for every driver.

**Vindication of orchestrator's leading hypothesis** (with refinement): the crash IS a COM apartment issue, but it manifests on the SECOND `/load_preset`, not the first. The first ASIO open silently "works" without COM init because no prior thread has corrupted ASIO state. The second open fails because the first worker thread exited and tore down ASIO with it.
