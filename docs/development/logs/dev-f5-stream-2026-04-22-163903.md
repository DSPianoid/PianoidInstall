### Revised interpretation ‚Äî 14:20 (after same-harness A/B)

The initial interpretation was wrong. analyse-distortion reported a kwarg typo in their A1 harness (`string_iterations` vs `string_iteration`) that silently defaulted to iter=12, not iter=8. So my "post-fix 33.4% vs pre-fix ~100%" was actually comparing iter=8 vs iter=12 workloads, not F5 vs no-F5.

Same-harness A/B (revert F5 ‚Üí rebuild ‚Üí measure ‚Üí restore F5 ‚Üí rebuild), same `Preset_test5` + `buffer_size=4` + SDL3 + 30s silent engine:

| Config | Pre-F5 | Post-F5 | Œî |
|---|---|---|---|
| string_iteration=8  | **33.3%** underrun, 13975 ¬µs max | **33.4‚Äì35.0%** underrun, 16311‚Äì18123 ¬µs max | within noise |
| string_iteration=12 | **110.9%** underrun, 18501 ¬µs max | **110.3%** underrun, 21031 ¬µs max | within noise |

**F5 has no measurable effect on silent-engine underrun rate.**

This refutes the specific hypothesis that default-stream `cudaMemcpy` + `cudaDeviceSynchronize` in `produce()` was a meaningful bottleneck. The synthesis kernel runs on the default stream; `produce()` runs on the same producer thread serialised by the host loop; the implicit dependency the kernel‚Üímemcpy serialisation created is only visible if the producer thread gets to its copy while the kernel is still running. Evidently the host loop structure already ordered these such that the kernel had completed before produce() was called, so the stream isolation removed nothing observable.

Remaining underruns are compute-bound (iter=12: kernel over budget) and jitter-bound (iter=8: kernel near budget, OS scheduling tips it over).

F5 is still architecturally better (producer copy should not implicitly wait on unrelated GPU work), so keeping it is defensible. But it is NOT a fix for the distortion symptom. Team-lead should know the ~100% ‚Üí ~33% drop initially reported was a measurement error, not a causal effect of F5.
