- In SDL2 those integers were 0-based ordinal indices into `SDL_GetNumAudioDevices()` — stable enough to use across runs in practice
- In SDL3 `SDL_AudioDeviceID` is `Uint32`, opaque, and CAN CHANGE BETWEEN RUNS (libsdl wiki: `SDL_AudioDeviceID.md` — "Zero is used to signify an invalid/null device", values are session-local handles, not persistent ordinals)
- **Decision: ADD a name-based resolution shim** during Wave 2:
  - When `RoomResponseRecorder.set_audio_devices` (or load_from_file) sees a non-default `input_device`/`output_device` integer, look up the *current* device list via `engine.get_input_devices()` / `get_output_devices()`
  - If no name companion is persisted, the integer is treated as a 0-based ordinal into the *current* device list and the resolved device's NAME is then persisted as `input_device_name`/`output_device_name` for next-run resolution
  - On subsequent runs, prefer name-based match; integer becomes a fallback ordinal only
- Limits: gracefully fall back to `-1` (default device) if name doesn't match, with a warning logged. Do NOT fail-hard on a missing device name — user may have unplugged the UMC1820 between sessions.
