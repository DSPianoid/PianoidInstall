**APPROACH (chosen over the overflow-Menu pattern, with rationale):**
The toolbar holds HETEROGENEOUS rich controls — two custom `<Slider>` widgets (VolumeSlider/FeedbackSlider, drag), a preset `<Select>`, the `BackendStatusIndicator` (own popovers), a shared `<NumInput>`. Re-homing those into a `<Menu>`/`<MenuItem>` overflow list (the brief's PREFERRED pattern, ideal for simple icon rows) would break their interaction (a Slider/status-popover inside a Menu is awkward + fragile). The brief explicitly allows "OR horizontal scroll (overflowX:auto) if the toolbar is an iconic row" — for a dense audio-tool toolbar of mixed real controls, scroll GUARANTEES zero functionality loss (every control stays a real, fully-interactive control) with a minimal, contained diff.
Fix = entirely inside `ToolBar.jsx`:
  (1) `<Toolbar>` gets `sx={{ overflowX: 'auto', flexWrap: 'nowrap' }}` + a thin dark-theme-styled scrollbar (WebKit + Firefox `scrollbarWidth/scrollbarColor`) → overflow is confined to the toolbar (page-level h-scroll gone), reachable by scroll/trackpad/shift-wheel at ANY width.
  (2) `flexShrink: 0` on the flex children groups (logo, left content via `gap`, spacer keeps flexGrow:1, right cluster) so labels/inputs/sliders never get internally squished — they keep full size and the row simply scrolls.
  (3) Wide layout: no overflow → no scrollbar shown → byte-identical to today.
