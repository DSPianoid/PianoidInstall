**Design (P1/P2 honored; gated DEFAULT-OFF; alert-not-kill; debounced one-per-event):**

- **Gate:** a single NEW opt `proactiveAlerts?: boolean` on `SessionHostOptions` (default false). OFF ⇒ (a) the in-flight watchdog is NOT enabled (session-host does not pass `turnTimeoutMs`/`onStallAction` to the lifecycle → the watchdog timers do not arm), (b) the proactive-watch scheduler does not start, (c) `maybeProactiveAlert()` early-returns → NO push. So OFF = byte-for-byte today (SACRED INVARIANT). The live index.ts does NOT set it → live behavior unchanged.
- **In-flight watchdog enablement (ALERT-not-kill):** when `proactiveAlerts` is on, the session-host passes `turnTimeoutMs: turnWatchdogMs (default 180000)` + `onStallAction: 'surface'` (NEVER 'restart') to the LifecycleManager ctor. The watchdog `surface` path fires `onStall` (already latches `lastStall`) — A5's `onStall` hook ALSO calls `maybeProactiveAlert()`. NO auto-kill/restart is ever introduced (the lifecycle 'restart' stall action is deliberately NOT used).
- **STUCK via missed ping:** in `pingLiveness`'s timeout callback, when `proactiveAlerts` is on, latch a `lastStall` (silentMs = the response timeout, action 'surface') + call `maybeProactiveAlert()` BEFORE the existing `opts.onUnresponsive?.()` — so `status` classifies STUCK end-to-end AND the alert pushes. The index.ts auto-restart still fires after (unchanged). Gated: with the flag off, no latch/alert is added here → byte-for-byte.
- **DEAD detection:** a dead child emits no events, so A5 arms a lightweight periodic `proactiveWatchTimer` (only when the flag is on) that calls `maybeProactiveAlert()` each tick; classify reads `lifecycle.health().running===false` → DEAD. Timer interval = `proactiveWatchIntervalMs` (default 20000), `.unref()`'d. The TIMER FACTORY is INJECTABLE (`proactiveWatchTimers?: {setInterval, clearInterval}`, default globals) so tests fire ticks synchronously via a fake — NO real wait, never hangs.
- **Debounce (one-per-event):** a field `alertedState: 'stuck' | 'dead' | null`. `maybeProactiveAlert()` computes `classifyLiveness(snapshot)`: if it is STUCK or DEAD and DIFFERS from `alertedState` → push ONE alert + set `alertedState`; if it is `active` → clear `alertedState` (RE-ARM, so the next stuck/dead event alerts again). Repeated ticks in the same state do NOT re-push (flood-safe). A stuck→dead transition pushes once for dead (distinct event).
- **Message:** states what's detected + what to do, e.g. STUCK: "⚠️ Orchestrator unresponsive ~Xs — tap /control → Restart, or it keeps retrying." DEAD: "🔴 Orchestrator died — restarting with handoff. Tap /control to check." (composes with, does not replace, the existing death/unresponsive auto-restart notice).
- **P1:** no new state owner. Lifecycle stays the sole stall DETECTOR; session-host stays the sole owner of `lastStall`/`alertedState`/the operator/the push path. **P2:** detection in lifecycle, classification pure in control-command (reused), push (channel I/O) in session-host. No module widens its concern. lifecycle.ts: ZERO structural change (A5 only USES the latent watchdog). config.ts: thresholds resolvers. control-command.ts: a tiny alert-text formatter (pure) so the message strings are unit-testable.
