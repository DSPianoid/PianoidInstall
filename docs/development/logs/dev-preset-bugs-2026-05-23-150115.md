#### ROOT CAUSE — Bug #1 (isolation leak): FRONTEND editor-history not re-initialised on preset switch + speculative re-emit (the dev-833f pattern)

`PianoidTunner/src/PianoidTuner.js`:
- Init effects for the per-pitch/per-mode editors are **init-once, guarded**:
  - strings (1059-1063): `if (!parametersOfStrings || stringsHistory?.values) return;`
  - modes  (1146-1150): `if (!parametersOfModes || modesHistory?.values) return;`
  - excitation (1224-1229): `if (!parametersOfExcitation || excitationHistory?.values) return;`
  Once `*History.values` is set, the effect NEVER re-inits — and it does NOT depend on
  `presetVersion`. So after a preset switch the editor history STILL holds the previous
  preset's edits. The displayed editor reads `*History.values`, so the user SEES copy#1's edits
  in copy#2.
- Speculative-emit effects (strings 1066-1132; modes 1152-~1205; excitation 1231-…) watch
  `*History.values` + `parametersOf*`. When a switch re-fetches fresh `parametersOf*`, these
  effects compare it against the STALE `*History.values` and **re-POST the divergence to the
  backend** via `changeParametersOf*`. → copy#2's backend state is actively overwritten with
  copy#1's edits. This is EXACTLY the dev-833f state-watching-useEffect re-push bug that
  SYSTEM_OVERVIEW "Frontend↔Backend State Discipline" #3 warns about. (Excitation already has a
  partial band-aid: `skipExcitationSyncRef` at 1233.)
- `useSoundChannels.js` is the CORRECT reference: it re-inits unconditionally on `[…, presetVersion]`
  (309-323) and has no speculative emit. The string/mode/excitation/feedin/feedback editors were
  the "deferred tech debt" called out in SYSTEM_OVERVIEW lines 113-116.
- FIX: re-init `stringsHistory`/`modesHistory`/`excitationHistory` from fresh backend data on every
  `presetVersion` bump (drop the once-only guard; add presetVersion to deps), AND suppress the
  speculative re-emit on the render where the change came from a preset transition (ref-skip,
  generalising the existing `skipExcitationSyncRef`). usePreset already clears SC caches +
  re-fetches + bumps presetVersion on switch (238-281) — the editors just aren't honouring it.

#### ROOT CAUSE — Bug #2 (intermittent switch): switchingRef silent-drop + spawn→switch chain

the last-requested name), OR keep guard but capture the desired target and re-run after the in-flight
switch settles. Will confirm exact path in live UI before finalising.

#### Edit 2: PianoidTuner.js — Bug #1 (editor-history isolation). Re-init on presetVersion + suppress speculative re-emit.
[EDIT] file=PianoidTunner/src/PianoidTuner.js
- Root design: all 5 editor histories (strings/modes/excitation via useValuesHistory; feedin/
  feedback via useMatrixHistory) were init-ONCE (guard `if (!data || history?.values/matrix) return`)
  and ignored presetVersion → stale edits leaked across preset switch; the speculative-emit effects
  then re-POSTed stale local state onto the switched-to preset (the dev-833f class bug).
- Fix (uniform, per editor): re-init the history when presetVersion differs from the last-seeded
  version (tracked in <editor>InitVersionRef, -1 initial) OR history empty. Keyed on presetVersion,

  transition render. Real edits / undo / redo leave the ref false → still propagate. Generalises the
  pre-existing skipExcitationSyncRef. Mirrors the blessed useSoundChannels (presetVersion re-init)
