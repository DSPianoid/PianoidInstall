## DIAGNOSIS

The bug is REPRODUCED in the live-engine path (REST `/play` + live audio loop running) but NOT in the offline-render path (`runOfflinePlayback`). This is consistent with the user's report ("bug doesn't reproduce without UI") and with dev-cflfix's prior finding that offline mode-level state is healthy.

**The regression is NOT in synthesis math.** It is in the live-engine state path — note-off events are not properly clearing mode forces / hammer state when the audio driver loop is running.

### Where the bug is NOT
- Synthesis math (offline confirms decay works)
- Mode-level engine state (dev-cflfix confirmed P57 mode counts decay correctly)
- Per-event delivery: `/playback_stats` reports `total_events_pushed=14, total_events_drained=14` — no dropped events
- VirtualPiano.js `handleMouseUp` (note-off pitch == note-on pitch) — I bypassed the canvas with REST, still reproduces

### Where the bug LIKELY IS
- The live audio loop in `pianoid_middleware.unified_engine` or `pianoid.live_play_loop` that consumes events and dispatches to CUDA — the note-off command is somehow not triggering the same damper-down path that offline render uses
- OR the runtime parameters that get applied at engine startup (Audio On + Listen To MIDI + listen_mode=Strings) configure the engine in a state where note-off becomes a no-op for mode forces
- OR the strings-listen-mode cascade (recently fixed on current dev) interacts with note-off in a way that leaves mode forces non-zero

### Suggested commits to bisect first (in PianoidCore fdf3dd2..67148fa range)
1. **dev-cflfix CFL guard work** — recent, touches kernel-level damping/CFL handling; matches the "synthesis math reduced ~3-5x" symptom; could affect runtime path differently from offline path
2. **dev-833f sound-channels work** — was the dominant change-set in this window; changed how mode forces are accumulated/cleared per-block; could leave residual force when note-off arrives mid-block
3. **dev-cal-twice + dev-cursor-drift** — touched live parameter cascade
4. **dev-3a08 length/dx work** — could change `dx` calculation in damper-down step
5. **dev-vpnoteoff** — NOT in dev branch; can rule out (we bypassed VirtualPiano entirely and still see the bug)
