**Bug summary (from evidence):**
- `handleMouseUp` (line 252): `const note = getNote(event)` recomputes pitch from the **mouseup position**
- Line 259: `onPlayNote?.({ pitch: note, command: 128 })` sends note-off for the RELEASE position pitch
- Line 260: `prev.filter((value) => value !== note)` removes the wrong entry from `notesPressedByMouse`
- If cursor drifts even 1 key between press and release, note-off goes to a neighbor → pressed note never released
- If released off-canvas, `getNote` returns `undefined` → NO note-off emitted at all → pitch rings forever
- `onMouseLeave` also bound to `handleMouseUp` (line 271) — same buggy path fires on canvas exit

**State tracking structure:**
- `mouseDownPitch` (useState, line 39): holds the pitch from `handleMouseDown`'s `getNote(event)` call (the pressed position)
- `notesPressedByMouse` (useState, line 40): array of currently-held mouse-pressed pitches; appended in `handleMouseDown` line 231

**Fix direction:**
- `handleMouseUp` must use the pressed pitches tracked in `notesPressedByMouse` (or `mouseDownPitch`) for note-off — NOT `getNote(event)`
- Release ALL entries in `notesPressedByMouse` on mouseup/mouseleave for the right button
- Clear `notesPressedByMouse` and `mouseDownPitch` after release

**Drag / glissando check:**
- `handleMouseMove` (line 236): only fires range selection (left-button drag), guarded by `event.button === 0`
- The commented-out line 248 (`//onPlayNote?.({ pitch: note, command: 127 })`) shows right-button drag was considered but disabled
- There is NO intentional right-button glissando behavior — dragging while right-button held is not a feature
- Safe to release the pressed pitch on mouseup/leave without breaking any drag feature

**P1/P2 check:**
- P1: `notesPressedByMouse` state is owned by this component. The fix makes the same owner write it correctly. No P1 violation.
- P2: `VirtualPiano.js` concern is rendering the piano and emitting play/select events. The fix stays entirely within that concern.
