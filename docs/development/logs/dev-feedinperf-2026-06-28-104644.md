### Step 5 (addendum): full MEASURED cost attribution — 2026-06-28T11:55:00Z
(team-lead asked to rule out backend + confirm FE-per-move before accepting the fix)
- **BACKEND, ruled out:** direct `POST /set_parameter/feedin/All` with the live full 84×196 matrix (332 KB) = **170–206 ms**, status 200, ×3. And `changeFeedInValues` DEBOUNCES the POST (usePreset.js:671) so only the last fires per gesture → ~0.2s ONCE per drag, not minutes. The code-map's "backend processes the bulk POST slower" guess is REFUTED.
- **FRONTEND per emit, dominant:** the harness stubs changeFeedInValues entirely, so 143,137 ms → 32 ms is provably 100% client-side (zero network) — ONE `applyAggregateChange` (fan-out + history apply + emit-staging), the quadratic full-matrix clone.
- **CADENCE = per mouse-move (team-lead hypothesis #2 CONFIRMED):** `DrawableChart.onMouseMove` calls `emitVector` on EVERY move (DrawableChart.jsx:471), not on mouseup → the 143s fan-out fired per processed move = minutes. Post-fix 32 ms/move → ~30fps responsive drag.
- **RE-RENDER:** averaged view renders ONE aggregate curve (88 pitches collapsed to a single DrawableChart), NOT 88 charts (that concern is the non-aggregate per-cell matrix view).
- **Why SC fast (same cadence):** collapses ~4 channel rows vs ~88 pitches → ~22× fewer sub-changes AND ~22× smaller clone per move → ~hundreds× cheaper. Pure collapse-dimension difference.
