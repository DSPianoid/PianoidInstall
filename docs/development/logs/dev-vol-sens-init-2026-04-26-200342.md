**KEY INSIGHT — the bug is a UI semantic mismatch, not a missing reset:**

C++ `calculateVolumeCoefficient()` (Pianoid.cu:1192) branches on volume_center:
- center==0 → **legacy** formula: coeff = max_volume^(level/127)
- center>0  → **new**    formula: coeff = center * range^((level-64)/63)

Frontend ToolBar.jsx VolumeSlider line 40: `center = exp((presetVolume + 64) / 8)`. When user
moves the sensitivity slider, applyRange() always sends `(c=positive, r=newRange)`. So:

- AT STARTUP: backend has (range=10, center=0) → **legacy** formula uses `max_volume` (preset-dependent, e.g. 5e18 for big presets)
- USER touches slider → frontend sends positive center → backend switches to **new** formula
- USER returns to slider=10 → frontend sends (c=positive, r=10) → backend uses **new** formula with r=10
- Round-trip transitions backend from LEGACY (loud) to NEW (10x range, much quieter / matches user expectation)

The slider UI label "10" means different things depending on whether `volume_center` is 0 or positive.
Backend operating value at startup ≠ what slider shows.

**Sensitivity computation with main_volume = preset_volume = 120 (UI default):**

| Level | Legacy coeff (center=0) | New coeff (center=9.7e9, range=10) | Legacy/New |
|-------|------------------------|-------------------------------------|-----------|
| 0     | 1.000e+00              | 9.395e+08                           | ≈0 |
| 64    | 9.745e+09              | 9.745e+09                           | 1.000 |
| 127   | 6.629e+19              | 9.745e+10                           | 6.8e+8 |

At startup (volume_level=64): both formulas give **same** coefficient.
But at any other level: **wildly different**. Legacy curve is steep (max_volume=6.6e+19), new
with range=10 is gentle (factor 100 from min to max).

**This is "sensitivity"** — the slope of coefficient vs slider position. Legacy mode is
extremely sensitive; new with range=10 is gentle. User moving slider away from 64 in legacy
mode → huge jumps. After round-trip → new formula → small jumps. Exactly matches the user
report: "Backend loads with much higher sensitivity. When I change it and then set back to 10
it becomes OK."

### Step 1 hypothesis confirmed (pending reproduce):
The "fix" on backend init isn't missing setter call — it's that the SEMANTICS of (center=0, range=10)
on backend differs from (center=positive, range=10) which is what the UI conveys.

**Likely fix:** Frontend should push initial (center=positive, range=10) to backend on first mount —
OR — backend should switch volume_center default from 0 to a positive value at construction so legacy
formula is never used unless explicitly opted into.

Recommend Option 3 (defense in depth): frontend pushes initial state, AND init_pianoid does the same
explicit reset that switch_preset does (set volume_center to a positive default to match new-formula
semantics implied by the UI).
