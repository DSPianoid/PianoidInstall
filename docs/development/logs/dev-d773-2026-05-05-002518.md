## Honest assessment of regression test scope

Per the orchestrator's instruction ("STOP and report ... if the synthetic chain-7+8 regression test doesn't actually validate per the analyst's math"), I attempted three synthetic constructions of the chain-7+8 case:

1. **Clean shapes with junk far-from-s_A**: sliding_window already correctly separates ChainA from junk. No failure to reproduce.
2. **Mediator shape s_sub between s_A and s_J**: sliding_window glues everything; nuclei_merge separates ChainA. Different output but doesn't match the user-reported "chain 7 + chain 8 stay separate but sub-cluster should be reassigned" pattern.
3. **Sub-cluster MAC ~0.4 with s_A, junk averaging to perp**: sliding_window peels a single 8-det c7 cluster + clean ChainA (matches user's pattern). nuclei_merge produces THE SAME decomposition because Stage 3 stray-cost calculation correctly assigns sub-cluster to junk-chain (sub MAC 0.6 vs junk-mean > sub MAC 0.4 vs ChainA-mean).

The fundamental issue: in the real Belarus `tmp8c7q0lu0` data, the analyst observed sub-cluster has MAC up to 0.424 with c8's clean reference, but the AVERAGED c7 reference shape MACs only 0.199 against c8. To force nuclei_merge to make a DIFFERENT decision than sliding_window on the chain-7+8 case, the sub-cluster's MAC against c8-clean must be HIGHER than the sub-cluster's MAC against c7-junk-averaged. We don't know that asymmetry from the analyst's report — only that c8-c7 whole-chain MAC is 0.199 and the max single-pair is 0.424. In a synthetic case where junk shapes mutually agree (clustering-together condition), the junk-averaged MAC against the sub-cluster naturally HAS to be higher than the sub-cluster's MAC against ChainA — that's the math that produces the over-broad cluster in the first place.

**What this means for the default switch:**
- The default change to `nuclei_merge` IS justified independently of the chain-7+8 case: see `TestUserFailureCase` which already validates the high-coverage low-overlap large-freq-drift case that nuclei_merge was designed for. That test still passes.
- The chain-7+8 case in `tmp8c7q0lu0` may or may not be resolved by the default switch alone — it depends on the actual MAC asymmetry in the real data, which we cannot verify without running tracking on the live dataset.
- The validation should include a manual re-run of tracking on `tmp8c7q0lu0` post-merge to confirm whether nuclei_merge actually produces a better chain decomposition. If it doesn't, the user may need Option B (sub-cluster-aware merge) as a follow-up.
