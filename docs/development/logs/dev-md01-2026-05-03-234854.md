- Probe: `D:/tmp/midi_dedup_probe.py` — repeatedly schedules NOTE_ON/NOTE_OFF via REST and WS, comparing `realtime_buffer.total_events_pushed` (visible via `/playback_stats`) before vs after each call. The dedup is upstream of `pianoid.schedule_event`, so a filtered duplicate produces +0 pushed events; an accepted duplicate produces +1.

| # | Action | events_pushed delta | Result |
|---|---|---|---|
| (a) | REST NOTE_ON 60 (call 1) | +1 | scheduled |
| (a) | REST NOTE_ON 60 (call 2, identical) | +0 | **filtered (works)** |
| (a2) | REST NOTE_ON 62 src=midi (1st) | +1 | scheduled |

| (c) | WS NOTE_ON 67 | +1 | scheduled |
| (c) | REST NOTE_ON 67 (cross-transport) | **+1** | **NOT filtered — BUG** |
| (d) | WS sid1 NOTE_ON 69 | +1 | scheduled |
| (d) | WS sid2 NOTE_ON 69 (different SID, same payload) | **+1** | **NOT filtered — BUG** |

- **Confirmed root cause** — the dedup state is split into two independent stores: `_ws_last_command` (per-SID, WS only) and `last_command` (module-global, REST only). When a duplicate crosses transports OR comes from a different WS SID, neither store catches it. The pre-WebSocket-era single filter (`last_command` only, REST only) caught everything because REST was the only ingress.
