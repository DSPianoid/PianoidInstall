## ★ SUPERVISOR RESTART PROCEDURE (investigated + documented; NOT executed — the orchestrator runs it)

**THE KEY QUESTION ANSWERED:** `POST /api/lifecycle/restart-request` does NOT reload the supervisor's `dist/`. It only cycles the orchestrator CHILD (the `claude -p` process) via `SessionHost.requestRestart`→`lifecycle.restartFresh()` (re-spawns the child, SAME SessionHost/LifecycleManager objects in the SAME supervisor process, keeping the supervisor's OLD in-memory code). The control-plane code lives in the SUPERVISOR (`session-host.ts`/`index.ts`/`control-command.ts`), so activation REQUIRES the SUPERVISOR PROCESS to re-spawn and reload `dist/`. (Authority: `D:\tmp\restart-supervisor.ps1` header — "a code change under tools/supervisor only takes effect when the SUPERVISOR PROCESS restarts — the in-session POST /api/lifecycle/restart-request only cycles the CHILD, not the parent" — + `launch-prod-orch.mjs:99-104` spawns `node dist/index.js` so a fresh process re-reads dist/; + `lifecycle.ts:511 restartFresh` only stops/restarts the driver.)

**EXACT COMMAND to reload the new supervisor dist/ (the orchestrator runs this):**
```
powershell -NoProfile -ExecutionPolicy Bypass -File D:\tmp\restart-supervisor.ps1 -Launcher prod
```
- **MUST run DETACHED** — YES, intrinsically. The script SELF-DETACHES via WMI (`Invoke-CimMethod Win32_Process Create`) on its FIRST invocation: it re-spawns itself parented to WmiPrvSE (NOT to the supervisor), so the subsequent `taskkill /F /T` of the supervisor tree (which hosts the orchestrator that launched the script) cannot take the script down mid-restart. The orchestrator can fire-and-forget it; the first call returns immediately ("dispatched DETACHED (WMI PID=…)") and the detached instance does the real work ~5s later. (Do NOT pass `-Detached` yourself — the script adds it on the self-respawn.)
- **What it does:** (1) self-detach via WMI; (2) `Start-Sleep 5s` (let the orchestrator's final channel message deliver before the channel is cut); (3) `taskkill /F /T /PID <owner of port 8790>` (stops the running supervisor + its claude child); (4) wait ≤20s for port 8790 to free (the launcher refuses to start if 8790 is owned — the single-instance guard); (5) re-run `launch-prod-orch.mjs` (re-reads the prod token from `~/.claude/channels/telegram/.env`, spawns a FRESH `dist/index.js --live --session --profile orchestrator --panel 8790` DETACHED+unref'd → loads the NEW dist/); (6) verify the new supervisor via `GET http://127.0.0.1:8790/api/health` (`started=true`) for ≤60s. Forensics → `D:\tmp\supervisor-restart.log`.
