1. **Create Project ignores user-provided name.** User types a custom name in the
   Create Project dialog, submits, and the project appears with the raw zip
   stem + numeric index (e.g. `PlyWoodTake1_5`) instead of the typed name.

   (mapping_config.json) should still be carried over because it describes the
   measurement setup, not the ESPRIT analysis.

## Bug 1 root cause

`ModalAdapter.create_project_from_zip` (lines 1772–1784) collides
**extraction-directory** name with **project name**:

```python
original_name = name
dest = os.path.join(extracted_root, name)
suffix = 1
while os.path.exists(dest):
    name = f"{original_name}_{suffix}"   # mutates project name!
    dest = os.path.join(extracted_root, name)
    suffix += 1
```

When the user-typed project name happens to share a stem with a previously-
extracted measurement folder (very common — re-importing the same source zip),
the extraction collision suffix is silently applied to the **project name**,
which then propagates into `create_project(name, ...)`, the on-disk project
folder name, and the user-visible `result["name"]`.

The `extracted_root/<name>` collision and the
`projects_base/<name>` collision are **independent concerns**:

- **Extraction dir collision** is an internal artifact (we keep extracted
  measurements on disk for re-averaging). When it collides, we should suffix
  the **extraction dir alone** so the new measurements land in a fresh folder,
  but the project keeps its user-chosen name.
- **Project name collision** (the projects-base check on lines 1869–1882) is
  the only place where the user-facing name should ever be auto-suffixed.

## Bug 2 root cause

`ModalAdapter.copy_project` (lines 1348–1380) only copies
`measurements/scenario_*.npy` and rewrites `project.json` — it omits
`modal_adapter/mapping/mapping_config.json` entirely. So while the copied
project does NOT carry ESPRIT/tracking/feedin artifacts (good), it ALSO loses
the channel-role / grid-layout / pitch-offset configuration that the user set
up for the measurements (bad — the user has to redo the mapping before they
can run ESPRIT).

The user's report ("settings locked and data processed") suggests they may
have been hit by an older code path that did a full `copytree` of the project
directory. The current code already produces an empty `modal_adapter/`
sub-tree, but it's missing the `mapping_config.json` carry-over.

The correct contract for "Copy From" is:
- Copy: `measurements/`, `modal_adapter/mapping/mapping_config.json`,
  any other measurement-derived metadata in `project.json` (sample_rate,
  num_channels, scenario_indices, measurement_source, band_config).
- Skip: `modal_adapter/esprit/`, `modal_adapter/tracking/`,
  `modal_adapter/feedin/`, `modal_adapter/output/`, `modal_adapter/export_text/`
  contents (the directories are recreated empty).
- Reset: `applied` flag (no `output/applied.json` carried over).
