### Diagnosis — Bug A (heatmap stale post-merge)

Mechanism (read from source):

1. `mergeChainsMany` (useChainEditor.js:209-246) mutates frontend `editedChains` only — does NOT call backend. Merge doesn't persist to `_tracked_chains` until user clicks Save.
2. After merge, `selectedChains[0]` (the merge target) is the SAME id as before merge.
3. `GridHeatmapInset` useEffect deps are `[chainId, getGridHeatmap]` (GridHeatmapInset.jsx:44). Both unchanged → no refetch.
4. Even if it DID refetch, backend `/modal/grid_heatmap/X` returns chain X's pre-merge detections (backend hasn't been touched).

So Bug A = double issue: cache-key invariance + backend lag. Fix is to source heatmap from `editedChains` (which has the merged data) instead of from backend, since merging is a frontend-state operation.

### Diagnosis — Bug B (stab diagram + chain data stale post-retrack)

Mechanism (read from source):

1. User clicks Run Tracking → `runTracking` (useModalAdapter.js:1140-1195) POSTs to backend, sets `stages.tracking.data = res.data`, calls `syncFromBackend()` which re-fetches `tracking_results` into `stages.tracking.data` again.
2. ModalAdapter.jsx init useEffect (line 232-237) is gated on `chainEditor.editedChains === null`. After previous tracking, `editedChains !== null` → no re-init → editedChains keeps OLD chains.
3. ModalAdapter.jsx stabilization fetch useEffect (line 244-250) is gated on `!stabilizationData`. After previous tracking, `stabilizationData` is set → no refetch → stabilizationData keeps OLD chains+points+unassigned.
4. ModalResultsView priority `chainEditor?.editedChains || stabilizationData?.chains || chains` → renders stale `editedChains`.

Same family as dev-md05's project-switch bug (state surviving boundary), different trigger (re-tracking on same project).
