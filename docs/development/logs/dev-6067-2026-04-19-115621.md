### Step 4: Root cause & fix plan — 12:00
Root cause in `useModalAdapter.js`:
- Mount-only `useEffect` on line 236 calls `checkServerHealth()` once.
- Mount-only `useEffect` on line 424 (guarded by `mountedRef`) calls `fetchProjects()` once.
- If the modal adapter server (port 5001) is started AFTER mount — either via the chip click or externally — `fetchProjects()` is never re-invoked. `ensureModalServer` polls for health but does NOT call `fetchProjects` on success.
- Result: project list stays empty until page reload.

Fix plan:
- Add a lightweight health-polling `useEffect` that runs while `serverRunning === false`, polling `GET /health` every 2 s. On the first successful response, set `serverRunning=true`. The cadence is cheap and stops as soon as the server is reachable.
- Add a second `useEffect` that watches `serverRunning`. On off→on transition, call `fetchProjects()` and then `syncFromBackend()` + `fetchScenarioInfo()` if a project ends up being open — mirrors the existing mount-time init logic at line 427.
- Keep change targeted — no refactor of existing mount effects or `ensureModalServer`.
