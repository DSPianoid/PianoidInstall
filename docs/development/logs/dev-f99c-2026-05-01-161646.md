### Root-cause analysis (Step 1 + Step 1b combined)

**Diagnosis chain:**

1. Frontend / live REST request includes `audio_driver_type: 0` (default per BaselinePreset1 / route default).
2. `backendServer.py:684-685`: `adt = int(data.get('audio_driver_type', 4))`; `audio_driver_type = adt if adt in [0,1,2,3,4] else 4`. So `0` is valid and passed through.
3. `pianoid.py:2543`: `audio_driver_type = kwargs.get("audio_driver_type", 0)`. Forwarded to `init_pianoid`.
4. `pianoid.py:2003` (`pack_initialization_params_for_cuda`): when `audio_driver_type == 0`, the else branch sets `init_params.audio_driver_type = -1` ("Use compile-time default"). PRINT: "Playing with default driver, audio_driver_type 0".
5. C++ `Pianoid::Pianoid` constructor (`Pianoid.cu:187-195`): unconditionally calls `AudioDriverFactory::createDriverWithType(...)` regardless of `audio_on`. With `driverTypeInt == -1`, `AudioDriverFactory.cpp:83-84` resolves to `ACTIVE_AUDIO_DRIVER`.
6. `AudioDriverConfig.h:28-29`: when `USE_ASIO_AUDIO` is defined (which it always is for Windows builds per `setup.py:238`), `ACTIVE_AUDIO_DRIVER = AudioDriverType::ASIO` regardless of whether SDL3 is also compiled.
7. So driver_type=0 becomes ASIO at C++ side.
8. `start_realtime_playback_unified` (`pianoid.py:1334`): `config.audio_enabled = True` HARDCODED — the deferred bug from WIP line 72 means even `audio_on=0` doesn't disable audio on this path.
9. `OnlinePlaybackEngine::run` (`OnlinePlaybackEngine.cu:67-68`): `if (config_.audio_enabled) pianoid_->startAudioDriver();`
10. `Pianoid::startAudioDriver` (`Pianoid.cu:1676-1684`): calls `audioDriver->init()` then `audioDriver->start()`.
11. `ASIOAudioDriver::init` (`ASIOAudioDriver.cpp:42-53`): if `asioDriver.Init()` returns false, **throws `std::runtime_error("ASIO driver initialization failed — no working ASIO device found")`**.
12. Exception propagates up the call chain. Backend's `long_running_procedure` runs in a separate thread (`backendServer.py:760`), so the exception becomes a `print(...) + traceback.print_exc()` and the thread exits. But before that, the thread's exception handling in `long_running_procedure` (`backendServer.py:574`) catches it: `print + traceback`.
13. Per the team-lead's report, even a crash earlier in init can leave `pianoid_loaded=False` because `init_pianoid` has its own swallow at lines 1885-1887 (`except RuntimeError: print + return`).
14. Why does the Flask process die hard rather than just leaving the engine in a bad state? The C++ exception likely bubbles up through pybind11 bindings; uncaught C++ exceptions in the wrong path can terminate the Python process. Specifically — when `audioDriver->start()` throws inside `startAudioDriver` (line 1678) which is called inside `startApplication` (line 1718), the exception escapes to Python. The thread's try/except in `long_running_procedure` *should* catch it, but if the audio driver thread has already started a callback or the exception happens during a CUDA kernel, the process aborts.

**Test fixture works because:** `tests/system/conftest.py:97-108` creates `pianoid_audio_off` with `audio_on=False, audio_driver_type=0`, but **never calls `start_realtime_playback`**. Tests use `runOfflinePlayback` or direct kernel invocation. The audio driver IS constructed (it's unconditional in the Pianoid C++ constructor), but `init()` is never called — so the ASIO `Init()` failure is never triggered.

**Fix surface (small):**

The bug is that `audio_driver_type=0` resolves to ASIO, which is the WORST choice for "no audio" intent. Two options:

- **Option A (reactive):** map `audio_driver_type=0` to "no driver" semantically — i.e. when the request is type=0 AND audio_on=0, do not start the audio driver at all. But this requires plumbing in C++.

- **Option B (preventive, smaller):** change Python-side `pack_initialization_params_for_cuda` to use the BEST AVAILABLE driver based on what's actually present, rather than the compile-time default which can be ASIO when ASIO hardware isn't installed. Pick SDL3 if available, ASIO only when explicitly requested.

- **Option C (cleanest, small):** pin `audio_driver_type=0` to mean "SDL3" (which is always available for Windows builds since `audio_driver` config defaults to "SDL3"). Drop the -1 sentinel.

Recommended: **Option C** (simplest). The user's stated test contract for `audio_driver_type=0` is "no driver" (per WIP-44 description), but the in-process fixture works because driver isn't STARTED. Live backend always starts the driver. So either we wire `audio_driver_type=0` to skip driver startup OR we pin it to SDL3 which doesn't require dedicated hardware.

Given the team-lead said the user labels this "a clear bug in Pianoid initialization path", and a 1-10 LOC fix is expected, I propose to map `audio_driver_type=0` → SDL3 explicitly in `pack_initialization_params_for_cuda` (matching what it would do via the SDL3 path at type=3).

**Note on Bug #3 cross-cutting check:** Bug #3 is about `audio_driver_type=3` failing with `missing strings_in_pitches` in the Pianoid constructor. That's a Python-side packing/dispatch error in `Pianoid.__init__`. INDEPENDENT of Bug #2 — different failure mode (TypeError on construct vs. runtime ASIO init). They share the SAME entry point (`/load_preset` route handler) but diverge at the constructor signature handling vs. the audio driver init.

### Fix proposal

**Surface area:** `PianoidCore/pianoid_middleware/pianoid.py` — function `pack_initialization_params_for_cuda` (lines 2001-2003, the `else` branch that handles `audio_driver_type=0`).

**Decision:** route `audio_driver_type=0` to **SDL3** explicitly (matching what type=3 does), instead of using the `-1` "compile-time default" sentinel which resolves to ASIO when ASIO is compiled but no ASIO hardware is present.

**Rationale:**
- The user's intent for `audio_driver_type=0` (per WIP-44 description and conftest naming) is "default / no driver assumed". The current behaviour ("whatever was compiled in") makes the choice non-deterministic across environments.
- SDL3 is always available for Windows builds (default per `build_config.json`'s `audio_driver: SDL3`).
- SDL3 doesn't require dedicated hardware — it can fail gracefully on machines without an audio device whereas ASIO `Init()` throws hard.
- This matches the existing `audio_driver_type=2` SDL auto-detect path (`pianoid.py:1984-1992`) which prefers SDL3.

**Diff (preview):**

```python
# pianoid.py:2001-2003
# OLD:
        else:
            print(f"Playing with default driver, audio_driver_type {audio_driver_type}")
            init_params.audio_driver_type = -1  # Use compile-time default

# NEW:
        else:
            # audio_driver_type==0 ("default"): pin to SDL3 because the
            # compile-time default (`-1` → ACTIVE_AUDIO_DRIVER → ASIO when
            # USE_ASIO_AUDIO is defined) crashes on machines without ASIO
            # hardware. SDL3 is always available for Windows builds and
            # fails gracefully without dedicated audio hardware.
            print(f"Playing with default driver (mapped to SDL3), audio_driver_type {audio_driver_type}")
            init_params.audio_driver_type = pianoidCuda.AudioDriverType.SDL3
            init_params.circular_buffer_chunks = 16  # SDL3 needs larger buffer
```

- Run_online's exception handler caught it; engine went to `exception=true, audio_driver_active=false`.
- First load worked because the SDL3 buffer drains successfully on cold start; re-init after active state fails because the runtime expectations are tighter.

**Fix implemented at `pianoid.py:2001-2009`:**
