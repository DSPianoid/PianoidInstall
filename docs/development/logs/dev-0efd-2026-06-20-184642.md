### Step 4 (design) — empirical probes of the live `claude` CLI — 2026-06-20T16:05:00Z
Ran 4 throwaway probes against the real `claude.cmd` (haiku, subscription, no meaningful spend; all cleaned up) to settle the enforcement-point design:
1. allow-listed `Bash` + `--permission-prompt-tool stdio` → NO `control_request`; the tool runs. (sawControlReq=false, tool ran)
2. a NON-allow-listed tool (`Write`) → DOES raise `control_request{can_use_tool}`. So the floor (PermissionRouter.decide→routeWhen) ONLY fires for tools NOT on `--allowedTools`. Since Bash/PowerShell ARE allow-listed (profiles.ts:242-243), the shell safety floor is effectively DEAD on the live cli-stream path — even for the orchestrator's OWN commands, not just bypass sub-agents. (Pre-existing finding; broader than the task framing but the fix below covers it.)
3. ORDERING: at the moment the `assistant` tool_use line is emitted, the Bash command has NOT yet executed (marker file absent); killing the child tree at that line PREVENTS execution. → the driver has a real pre-execution interception window.
4. SUB-AGENT identity: the orchestrator's own `Agent` spawn line carries `parent_tool_use_id=null, subagent_type=undefined` (and the relaunch may be in `input.prompt`); the SUB-AGENT's `Bash` line carries `parent_tool_use_id="toolu_…"` + `subagent_type="general-purpose"`. NO control_request for either (Agent allow-listed; sub-agent bypass). → the driver CAN see + identify the sub-agent's relaunch Bash in the stream.

DECISION: enforce in the cli-stream driver's stdout loop (the ONE universal chokepoint that sees every tool_use regardless of permission mode / allow-list / bypass / background), BEFORE the line-130 sub-agent drop. Scan each `assistant` tool_use for a relaunch-family command (Bash/PowerShell command OR Agent/Task prompt matching the hardened isSupervisorRelaunchCommand); on a match invoke an injected guard + HARD-BLOCK by killing the child tree (proven to prevent execution). This is mode-independent → it cannot be defeated by bypassPermissions / run_in_background, which is the bug. The PermissionRouter floor (routeWhen) is NOT a reliable layer for shell tools (probe 2) so the predicate hardening is belt-and-suspenders for the SDK-driver / non-allow-listed path; the driver guard is the real fix. Hard-deny is task-blessed; the legitimate host-restart path is the lifecycle API (POST /api/lifecycle/restart-request / ctl:restart), NOT a raw shell relaunch, so blocking a raw shell relaunch in ALL cases is correct.
