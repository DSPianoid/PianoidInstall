# dev-bandui — Modal Adapter Band Configuration advanced-table population regression

**Started:** 2026-05-07T12:06Z

**Files locked:** `PianoidTunner/src/hooks/useModalAdapter.js`,

## User report

> "Bug from yesterday: I open Modal Adaptor / Band configuration / advanced,
> table is not populated"

Yesterday (dev-esrt 2026-05-06, PianoidTunner commit `8a0f764`) added a
hydration back-fill helper (`backFillBandDefaults` / `backFillBandsList`)
that joins the saved `esprit_config.bands` (which can lack newly-added
keys like `start_fade_ms`, `end_fade_ms`, `ir_length_ms`, `skip_start_ms`)
to the matching `extended_8band` preset entry by `name`, copying defaults
into any missing keys. That fix was confirmed working at the time. Today
the user reports the bug returned after a hard browser reset.

## Live evidence (chrome-devtools, before any code change)

Backend `/modal/band_presets` returns the full schema with all 4 fade
keys non-null on the Ultra-Low / Low rows — backend serializer healthy:

```
extended_8band[0] = {
  decimation: 8, end_fade_ms: 20, exp_factor: 0.15, f_max: 100, f_min: 30,
  filter_order: 4, ir_length_ms: 600, model_order: 8, name: 'Ultra-Low',
  skip_start_ms: 40, start_fade_ms: 20, window_length: null
}
```

`PlyWoodTake1_5` project has empty `esprit_config = {}` on disk — table
populates correctly there because the syncFromBackend `else` branch reads
defaults from `bandPresetsRef.current` (which IS populated by then in the
fast-network case).

`Belarus8D_c` project has a saved `modal_adapter/esprit/config.json` from
before the fade fields existed — its bands carry only
`{decimation, exp_factor, f_max, f_min, filter_order, model_order, name, window_length}`.
On load, the back-fill helper SHOULD restore the four missing keys from
`extended_8band`. Under fast-network conditions it does (verified live).

**Regression reproduction (deterministic):** I injected a 4 s artificial
delay on `XHR /modal/band_presets` via Chrome init script. Then loaded
the frontend with `Belarus8D_c` open. After the project_state response
returned (~1.5 s) the React `espritConfig.bands` had:

```
firstBandKeys: [decimation, exp_factor, f_max, f_min, filter_order,
                model_order, name, window_length]   (8 keys, NO fade keys)
allBands: [Ultra-Low, Low, Low-Mid, Mid, Mid-High, High, Upper, Top]
  (each: hasStartFade=false, hasEndFade=false, hasIr=false, hasSkip=false)
```

Even after the band_presets fetch eventually returned (5 s mark), the
state remained un-back-filled — the band_presets-load `useEffect`
short-circuits with `prev.bands.length > 0 ? prev : ...` and never
re-hydrates an already-populated bands list.

## Root cause

Two cooperating defects in `useModalAdapter.js`:

1. **Race between two mount-time `useEffect`s:**
   - L665 effect: `fetchProjects()` → `await syncFromBackend()` — this
     reads `bandPresetsRef.current` to build hydrated bands (line 597
     `backFillBandsList(ec.bands || [], presetName, bandPresetsRef.current)`
     and line 610 `bandPresetsRef.current[DEFAULT_CONFIG.preset]`).
   - L678 effect: `axios.get('/modal/band_presets')` → sets
     `bandPresetsRef.current = res.data`.
   - Both fire on mount. There is **no ordering guarantee** — whichever
     network response returns first wins. When band_presets is the
     loser (slower server processing, more requests in flight, slightly
     larger response, etc.), `syncFromBackend` runs back-fill against an
     empty `bandPresetsRef.current = {}` — `backFillBandsList` returns
     bands unchanged at line 107 (`if presetBands.length === 0 return`).

2. **No re-hydration after band_presets arrives late.** The L690
   `setEspritConfig((prev) => { if (prev.bands.length > 0) return prev; ... })`
   guard is intentional for the case "we already have bands from the
   server, don't overwrite with raw defaults". But it incorrectly
   short-circuits the re-hydration path: after the late arrival of
   `bandPresets`, we KNOW bands need a back-fill pass against the
   newly-cached preset data, but we skip it entirely.

In normal local-dev conditions the band_presets fetch usually wins
(~88 ms vs project_state's ~90 ms response), but the margin is so thin
that GC pauses, browser request prioritisation, or any backend latency
flip it. Once it flips, the bug is sticky for the whole session — only
explicit user action (changing band preset, hand-editing a band) clears
the bad state.

The reason the bug looked fixed yesterday and broken today is timing
luck — nothing in dev-bands or dev-robust changed the hydration path.
The race has always been there since dev-esrt landed; the user happened
to hit the losing side of it today.

## Fix plan

The minimal correct fix is to make hydration deterministic regardless
of which effect's HTTP call returns first. Two options:

**A. Sequence band_presets BEFORE the project sync.** Move the
band_presets fetch into the same mount effect as `fetchProjects` and
`syncFromBackend`, awaiting band_presets first. Pros: simplest;
guarantees ordering. Cons: serialises two independent network calls
that could otherwise be parallel; small wall-clock cost (~80 ms).

**B. Re-hydrate when band_presets arrives.** Keep the parallel fetches
but in the band_presets useEffect, after caching the new preset map,
call `backFillBandsList` against the CURRENT `espritConfig.bands` and
patch any missing keys. Pros: no serialisation, no cost on the win-side
path. Cons: more complex; needs to thread through current bands.

**Choice: A.** The ordering cost is ~80 ms once at mount, well below
the project-state request itself. Simplicity beats marginal latency
here, and removes the entire class of race bugs at this seam.
Additionally, I'll narrow the L690 guard so it doesn't unconditionally
no-op — keeping it as a safety net for the edge case, but no longer
relied upon for correctness.

Concrete edits:

1. Merge the band_presets fetch into the mount effect, before
   `fetchProjects` / `syncFromBackend`. Set `bandPresetsRef.current`
   synchronously from the response so any subsequent reader sees it.
2. Remove the now-redundant separate band_presets useEffect.
3. Add tests for `backFillBandDefaults` / `backFillBandsList` covering:
   - back-fill restores missing fade keys from extended_8band when
     saved bands lack them (the dev-esrt scenario, regression-locked).
   - explicit `null` is preserved (user opt-out).
   - bands with names not in any preset entry pass through unchanged.
