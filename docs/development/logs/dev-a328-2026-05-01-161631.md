**Key findings:**

1. **No `Settings.jsx` component exists.** The Settings pane is a `switch` case that renders `<ObjectInspector>` against whichever PropertyManager is currently selected. `ObjectInspector` is *generic* — it works for ANY of the 9 settings buckets. The "monolithic Settings panel" is really a *single-instance generic inspector* that happens to display whichever bucket is currently set in `currentPropertyManagerName`. There is no "preset section + per-panel section" inside it — it shows ONE bucket at a time, switching as the user clicks gear icons.

2. **Initialization-parameter / panel-local-settings split is already implicit.** The 9 buckets in `useSettings.js` are:
   - `presetLoadSettings` — initialization parameters (path, audio_driver_type, sample_rate, debug_mode, listen_to_modes, sound_derivative_order, etc.) — 14 keys
   - `virtualPianoSettings` — Virtual Piano panel local settings
   - `modesSettings`, `stringsSettings`, `chartSelectorSettings`, `feedInSettings`, `feedbackSettings`, `soundChannelSettings`, `workbenchSettings` — panel-local settings for each pane
   So the user's "two concepts" already exist as separate buckets — they're just BOTH rendered through the same Settings pane.

3. **Preset save / library management already exist in the top ToolBar** (SaveAsIcon, FileUploader, library Select dropdown, +/− buttons). They are NOT in the Settings pane. The user's request to "put preset saving and library management" into the Preset panel means: move those toolbar controls (or duplicate them) into the new Preset panel, presumably with a richer UI than the current Save-As prompt.

4. **The per-panel settings popup pattern already exists in ModalAdapter.jsx** — gear icon portaled into `.mosaic-window-controls`, click toggles a Collapse/Paper. This is a proven pattern in this codebase. The plan should re-use it (or generalise it) rather than invent a new one.

5. **PianoidTuner.js is RED (2496 LOC) and usePreset.js is RED (1584 LOC).** Per CODE_QUALITY.md C4, this refactor must NOT add to PianoidTuner.js without a split plan. Extraction is required — net LOC delta on PianoidTuner.js should be NEGATIVE.

6. **Mosaic layout migration:** `useLayout.js:22` hardcodes `"Settings"` as a leaf in `initialMosaicLayout`. Existing users have `"Settings"` saved in their `localStorage.mosaicLayout`. We need a `stripDeprecatedPanes` (analogous to the existing `stripDynamicWorkbenches`) that maps old `"Settings"` ID → new `"Preset"` ID on first load. Otherwise existing users will see a "Window content not found" message.

7. **State-discipline check:** The current Settings flow uses a small useEffect at line 944 that re-initialises the PropertyManager whenever `currentPropertyManagerName` changes by reading from a `settingsMap`. This is a single-source-of-truth read (settings buckets are the source) → not an anti-pattern. The Preset pane's load-on-presetLoadSettings-change useEffect (line 907) IS a speculative-emit pattern (writes settings → triggers backend load via useEffect) but it's an entrenched contract used to avoid double-fire. We will preserve it as-is (it's outside the H3 anti-pattern's scope — it's a one-way settings → APPLY flow, not a state-survives-across-events flow).

8. **MUI v6 dialog choice (per context7 lookup):** Dialog/Drawer/Popover all build on Modal. Dialog is the canonical fit for "settings panel with form fields + Apply/Restore" because (a) it's modal (forces attention on settings, prevents stray backend writes during edits), (b) it has DialogTitle/DialogContent/DialogActions structure that maps directly onto ObjectInspector's title + body + buttons, (c) MUI handles focus trap + Escape + scrim + portal automatically, (d) the existing project already uses `<Dialog maxWidth="md">` for the Window Layout Manager (line 2397) — same visual language. Drawer is wrong (too wide, side-anchored, would dock to a screen edge — Pianoid is mosaic-tiled, no natural edge). Popover is wrong (anchored, cramped for a 14-field form, would be clipped by the mosaic pane bounds).
