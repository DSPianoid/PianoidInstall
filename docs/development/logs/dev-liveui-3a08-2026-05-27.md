## [PROGRESS] ROOT CAUSE FOUND — frontend note-off targets the RELEASE position, not the pressed pitch
(team-lead redirect: instrument what the FRONTEND actually emits over the wire, via the REAL browser UI.)

### How notes are emitted (corrected): NOT plain fetch.
`playNote` (usePreset.js:1291-1318): tries Socket.IO first — `socketEmit('play', payload)` — and `return`s if sent;
REST fallback is `axios.post('/play')` (XHR, not window.fetch). My initial window.fetch hook saw nothing because
play goes over **Socket.IO**. Hooked `WebSocket.prototype.send` to capture the literal socket.io frames = ground truth.

### Ground-truth emission frames (real canvas press/release on the Workbench VirtualPiano, 84 keys, ~4.94px/key):
- SAME-position press+release pitch 56:
    ON  42["play",{"pitch":56,"velocity":127,"command":144,...}]
    OFF 42["play",{"pitch":56,"velocity":100,"command":128,...}]   ← correct pairing
- PRESS 56, RELEASE drifted +2 keys (idx35):
    ON  pitch 56 / command 144
    OFF **pitch 58** / command 128   ← NOTE-OFF GOES TO THE WRONG PITCH; 56 never released
- PRESS 56, RELEASE drifted +1 key (idx34):
    ON  pitch 56 ; OFF **pitch 57**   ← wrong pitch again
- PRESS 56, RELEASE off-canvas (left):
    ON  pitch 56 ; OFF **(no valid note-off emitted)**  ← getNote→undefined; 56 stranded entirely

### Root cause (VirtualPiano.js handleMouseUp, lines 252-262):
    const note = getNote(event);              // recomputes pitch from the MOUSEUP position
    ...
    onPlayNote?.({ pitch: note, command: 128 });   // note-OFF uses release-position pitch, NOT mouseDownPitch
handleMouseDown (line 230) correctly uses the press-position `note` for NOTE_ON and tracks it in
notesPressedByMouse (231); but handleMouseUp releases `getNote(mouseup)` (259) and filters notesPressedByMouse by
that wrong value (260). On a ~5px-wide key, ANY small mouse drift between press and release (normal human motion)
sends the note-off to a neighbouring pitch — or drops it if released off the keys — so the PRESSED note never gets
its note-off → rings forever = **"56 doesn't decay on note-off."** A spurious note-off also damps a pitch the user
didn't release. REST /play BYPASSES this (it always pairs on/off by explicit pitch) — which is exactly why the
buffer "looked fine" via /play but the real UI is broken. This is the UI-specific defect the team-lead suspected.

### FIX (for follow-up /dev — NOT this pass): VirtualPiano.js:259 must use `mouseDownPitch` (the pressed pitch),
not `getNote(event)`. Likewise line 260 should filter by `mouseDownPitch`. Also clear `mouseDownPitch` in the
non-left branch. (Frontend-only, no CUDA rebuild.)

### Relation to the earlier audio findings:
This note-off-mismatch is the dominant cause of the live "stuck/doesn't-decay" symptom under real mouse play —
distinct from, and more directly user-facing than, the engine-side mode accumulation. The attack over-gain/Sint32
overflow and the 57/59 near-silent dropout remain separate engine-side issues. Three independent defects stand;
the team-lead's frontend-emission hypothesis is the one that explains the real-UI "doesn't decay."
