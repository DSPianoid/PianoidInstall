**Architecture findings:**

1. **Launcher does NOT call /load_preset.** `PianoidTunner/server/launcher.js` only spawns processes (frontend, backend); the React frontend `usePreset.js:128` is the actual `/load_preset` caller, using the user's saved settings from `useSettings.js`. Therefore "default at the launcher" = (a) flip the React `DEFAULT_SETTINGS.listen_to_midi` from 0 → 1, AND (b) add an environment-variable override at the backend so a sysadmin / test-harness can force the value regardless of client.

2. **emit_midi_callback already wired (Phase 0).** `backendServer.py:796` injects `emit_midi_note_event` into `init_kwargs`; `pianoid.py:285` stores it; `pianoid.py:1546–1550` calls it inside the listener loop AFTER `schedule_event`. The callable performs `socketio.emit('midi_note_event', {...})` at `backendServer.py:580`. Already firing whenever the listener thread is alive (Phase 0 was tested in W2).

3. **Listener lifecycle.** `self.listen = True` + `_midi_thread = threading.Thread(target=MIDI_listener_unified)` is started by `start_midi_listener_unified()` (pianoid.py:1467). The loop exits cleanly on `self.listen = False`, closes the rtmidi port, sets `_midi_active_port = None`. `stop_playback()` at pianoid.py:382 already does this stop-and-join. So a runtime POST /midi/stop is essentially `self.listen = False; thread.join(timeout=1.0)`. POST /midi/start can call `pianoid.start_midi_listener_unified(midi_port)` when the playback is active and no listener is running.

4. **File-size watch (C4).** backendServer.py = 3246 LOC (RED), pianoid.py = 2773 LOC (RED). Phase 2 adds ~50 LOC of REST endpoints and ~10 LOC of env-var read; no helpers extracted. This is incremental on already-RED files — minimal additions. (Pre-existing G O entries already track these in CODE_QUALITY.md.)
