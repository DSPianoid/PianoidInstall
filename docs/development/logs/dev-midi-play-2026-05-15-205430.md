- P1 (Authority): the `if pianoid.listen: return` gate makes the play handler READ `pianoid.listen` (owned by the listener lifecycle) to decide play-path admission. `listen` is NOT the authority for "may a UI play command proceed." ### Double-trigger analysis (team-lead concern) ‚Äî 2026-05-15T18:25:00Z
- Team-lead asked: prove gate removal does NOT cause UI-play + hardware-MIDI to double-fire.
- `MIDI_listener_unified` reads bytes ONLY from the rtmidi hardware port (`listener.get_message()`, pianoid.py:1577). It does NOT observe `play`-handler events ‚Äî no echo loop.
- `add_realtime_event` (pianoid.py:1658-1697) pushes each event to `RealTimeEventBuffer` with NO dedup and NO cross-source coupling. One `schedule_event` call -> exactly one buffered event.
- A single UI play action -> one event. A single hardware keypress -> one event. The only way to get two notes is two distinct physical actions (hardware key + UI click) ‚Äî which is correct behavior, not a bug. The legacy gate did not "prevent" that case; it disabled the UI entirely.
- The per-source dedup (`_ws_last_command` per-SID WS / `last_command` module-global REST) handles the genuine duplicate (cross-transport WS->REST fallback of the SAME UI action) and is independent of the listener. VERDICT: gate removal is safe, no double-trigger risk.
