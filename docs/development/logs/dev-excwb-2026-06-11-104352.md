### Step 1: Understand Context — 2026-06-11T11:02:00Z

**How Strings/Modes get a Workbench (the established pattern):**
- Each param row in `Strings.jsx`/`Mode.jsx` renders a `<BarChartIcon>` IconButton → `onOpenWorkbench(key, value)` (spawn a NEW dynamic Workbench) AND `onActivate(key, value)` on row mousedown (pin the DEFAULT Workbench).
- In `PianoidTuner.js`: `onActivate` → `updateDefaultWorkbench(param, isPiano)`; `onOpenWorkbench` → `handleOpenWorkbench(param, isPiano)` → `openWorkbench` (useCurrentValues, builds id `Workbench:<groupe>.<name>...`).
- The Workbench (default + dynamic) computes its per-axis vector via `computeWorkbenchValues(wb)` and emits painted-vector edits via `onValuesChange → handleVectorChange(newVector)`. The WorkbenchFunctionTools cluster ALSO emits via `handleVectorChange` (renderToolbarControls 2554-2557). So fixing the Excitation branch of `handleVectorChange` covers default WB + dynamic WB + function-tools for free.
- `handleVectorChange` routes by `selectedParameter.groupe`: Strings/Excitation → `pitchesVectorDrawn`; Modes → `modesVectorDrawn`; into the respective history's `applyChange`. `useValuesHistory.calcChange` `pitchesVectorDrawn` zone does a FLAT per-pitch write `newValues[pitch][parameter]=value`.

**Excitation panel parameters (enumerated):**
- HAMMER params (flat per-pitch scalars, stored `values[pitch][param]`): `hammer_width`, `hammer_sharpness`, `hammer_position` — activated in `ExcitationProperties.jsx` row `onMouseDown` → `onActivate(name, null,null,null, value)`.
- GAUSS params (per level + per chart 0-4, stored `values[pitch][level][chart][param]`): `mu`, `sigma`, `shift`, `volume` — activated in `GaussEditor`/`GaussCell` (chart title click, cell click) + `GaussChart` chart select → `onActivate(name, levelValue, levelCaption, chartIndex, value)`.

### Diagnosis — 2026-06-11T12:12:30Z
- ROOT CAUSE (GENERAL, not Excitation-specific): `renderToolbarControls` (PianoidTuner.js:2712) only pushes the Close button `if (!isFullscreen)`. When ANY pane is maximized (`isFullscreen===true`), EVERY pane's title bar shows only the Restore button — no Close (X). The user noticed it on Excitation workbenches but it affects Strings/Modes/all panes equally.
- WHY it was suppressed: `handleMaximize(id)` sets `layout = id` (single-leaf fullscreen) + saves `layoutBackup`. A naive `mosaicActions.remove(path)` on a maximized single-leaf layout empties the layout AND leaves `isFullscreen=true` + the stale backup → broken empty stuck-fullscreen state. So Close was simply hidden rather than made safe.
- FIX (general, all panes): render a Close button in fullscreen that does the SAFE thing — remove THIS pane's leaf from `layoutBackup`, restore that pruned backup, exit fullscreen. = "close while maximized returns you to the multi-pane view minus this pane". Reuses the existing handleRestore/layoutBackup mechanism; no fork; doesn't touch Strings/Modes wiring (same code path serves them).
