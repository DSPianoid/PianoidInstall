**Symptom:** "ASIO driver fails to load although ASIO-capable device is connected."

**Root cause (ENVIRONMENT/CONFIG — NOT a Pianoid code bug):** No ASIO driver is

`InitAsioDriver()` reads the (empty) ASIO COM registry via `asioDrivers->getDriverNames()`,
gets zero drivers, prints "ASIO: No working ASIO driver found!", and `ASIOAudioDriver::init()`
throws. The code is behaving CORRECTLY (fail-fast on a missing driver) — there is simply no
ASIO driver to load. ESI GIGAPORT is even the device the codebase expects

**REMEDY (environment, the actual fix):** install an ASIO driver for the GIGAPORT eX:
  (a) PREFERRED — ESI's native ASIO driver for the GIGAPORT eX (vendor download; lowest
      latency, proper multi-channel). After install, HKLM\SOFTWARE\ASIO gains the driver's
      CLSID entry and adt=4 will enumerate + load it.
  (b) FALLBACK — ASIO4ALL (https://asio4all.org) — a generic ASIO wrapper over WDM. Works
      with any device but adds a thin software layer.
Until an ASIO driver is installed, the correct working config is SDL3 (adt=3) or SDL
auto-detect (adt=2) — which is what the user's current backend is already running (and it
has sound: /health audio_driver_active=True on the SDL3 path).

**Why the user already has sound (no crash):** the live backend's constructed driver is SDL3
(runtime log). Since createDriverWithType has NO ASIO→SDL fallback in C++ (a failed ASIO
init throws and leaves audio_driver_active=FALSE — NOT a running SDL3), SDL3 being healthy
proves the effective adt on the running session was SDL (2 or 3), not 4. So either the user
already switched the Settings panel to SDL as a workaround, or this load used a non-4 value.

**OPTIONAL code follow-up (NOT implemented — needs user/team decision; flagged only):**
There is a UX gap: if a user keeps the default adt=4 (ASIO_CALLBACK, per useSettings.js) on a
machine with no ASIO driver, `startAudioDriver()` throws and the engine ends with
audio_driver_active=FALSE → NO SOUND, until they manually switch to SDL. The documented
fallback chain (ASIO→SDL3→SDL2, STARTUP_TROUBLESHOOTING.md) is a MANUAL recovery, not
automatic. A graceful auto-fallback (catch the ASIO init throw → reconstruct as SDL3 →

because: (1) root cause is environmental; (2) it changes driver-selection semantics (a

requested one fails?); (3) the task is "ASIO fails to load," whose correct remedy is
"install the ASIO driver," not "silently use another driver." Reporting to team-lead for a
