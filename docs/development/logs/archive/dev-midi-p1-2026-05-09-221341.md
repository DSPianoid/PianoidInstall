#### Bug summary (from kernel-midi-batch-investigation-2026-05-08.md, confirmed by source read)
- `OnlinePlaybackEngine::processEventsAtCycle` correctly drains `all_events` for one cycle (rt + scheduled).
- But each `dispatcher_->dispatch(event)` for NOTE_ON calls `PlaybackCycleExecutor::exciteStringsForPitch` which OPENS+CLOSES its own `beginStringBatch()/commitStringBatch()` envelope, overwriting the prior commit's `noStrings_in_GP` and `new_notes_ind`.
- Last-event-wins behavior for chord-class workloads (4-note chord -> only last note's strings excited).
- Same bug reproduces in `OfflinePlaybackEngine::processEventsAtCycle` -> `PlaybackCycleExecutor::processEvents`.
- TEST_STRING_ONLY also broken: calls `addOneString` which itself does begin+commit.
- TEST_MODE_ONLY also broken: `handleTestMode` calls `addModeExcitation` then `commitStringBatch()` directly mid-loop.

#### Fix design (Option A from investigation — user-confirmed)
1. Lift `begin/commit` envelope from per-event to per-cycle.
2. In `OnlinePlaybackEngine::processEventsAtCycle` and `OfflinePlaybackEngine::processEventsAtCycle`:
   - Cap `all_events.size()` at `MAX_EVENTS_PER_CYCLE = 256`; record overflow in stats.
   - Always wrap dispatch loop with `pianoid_->beginStringBatch()` ... `pianoid_->commitStringBatch()`.
   - Preserve all_events insertion order (rt first, then scheduled — current behavior).
3. Rewrite handlers to STAGE-only (no commit):
   - `handleNoteOn`/`handleNoteOff` -> call `stageStringsForPitch` (new helper in PlaybackCycleExecutor — exciteStringsForPitch minus begin/commit).
   - `handleTestMode TEST_STRING_ONLY` -> use `stageStringsForPitch(target_index_as_pitch_proxy)` OR new `stageOneString` helper (still need to research; the current test-string path uses `addOneString` directly with target_index, not pitch).
   - `handleTestMode TEST_MODE_ONLY` -> use `addModeExcitation` (already a STAGE primitive — pending_mode_excitation_index). REMOVE the explicit `commitStringBatch()` call. The cycle-level commit fires the staged mode (commitStringBatch already drains pending_mode_excitation_index).
4. Add `MAX_EVENTS_PER_CYCLE = 256` to `constants.h`.
5. Add `dropped_events_per_cycle_overflow` to `OnlinePlaybackEngine::EngineStats`.
6. Rewrite `dispatchBatch` to be the actual entry point (not just a wrapper that loops `dispatch`). It takes the event vector, applies the envelope, calls the staging dispatchers, commits.
   - Or alternatively, keep `dispatchBatch` with the envelope behavior so engines can call it cleanly.

#### Open questions / inferred-only facts to verify in DMC
- TEST_STRING_ONLY uses `target_index` directly (uint16_t string index), not pitch. So `stageStringsForPitch(pitch)` won't work — need `stageOneString(stringNo, velocity)` that calls `_add_string_for_playback` without the begin/commit. — DMC must cite this.
- `commitStringBatch` MUST be called even with `noStrings_in_GP == 0` if a mode is staged (otherwise pending_mode_excitation_index never gets flushed).
