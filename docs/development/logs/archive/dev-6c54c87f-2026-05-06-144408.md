# Follow-up: dev-6c54c87f decka-fix (2026-05-06 18:00)

**Bug reported by user:** "In exporting tool: decka_coeff filled with zeros"

**Reproduction:** Ran `POST /modal/projects/PlyWoodTake1_4/export_text`
against the live backend; `decka_coeff.txt` is `0 / 2048 non-zero`
(= entirely zero). The other 4 text files are populated correctly.

**Root cause analysis:** TWO bugs, both in the `decka_coeff` packing path.

### Bug 1: empty `channel_to_sound` makes the whole loop a no-op

PlyWoodTake1_4 has `channel_roles` populated (1 force + 7 response) but
`channel_to_sound = {}` (empty — the user ran tracking but never visited
the Apply panel's "Sound output mapping" sub-section). My code did
`channel_to_sound.get(ch_idx)` and got `None` for every channel, so no
amplitudes were ever written.

### Bug 2: amplitude-array indices were treated as measurement-channel indices

`esprit_runner.py:194` pre-filters signals via
`signals[:, response_channels]` BEFORE handing to ESPRIT. The resulting
`mode_shape` is therefore `(M, len(response_channels))`-shaped — index `0`
is the **first response channel**, NOT measurement-channel `0`. My
original code treated `ch_idx` (0..N-1 in the amps array) as a
measurement-channel index, so for PlyWoodTake1_4
(`response_channels = [1..7]`) it tried to look up
`channel_to_receiver[0]` which mapped to nothing — silently dropping the
first response channel and shifting everything down by one. Even with
the empty-`channel_to_sound` fallback in place, this off-by-one would
still produce only 6 of 7 response channels in `decka`.

The same off-by-one was present in `extract_signed_shapes_per_chain`'s
mask logic: `for ch in response_channels: if 0 <= ch < n_channels:
mask[ch] = True` produced a 6-of-7 mask (instead of all 7) for the
pre-filtered case.

### Fix

`pianoid_middleware/modal_adapter/external_export.py`:

1. **Channel-to-receiver fallback:** when `mapping.channel_to_sound` is
   empty, derive a contiguous mapping from `mapping.response_channels`:
   first response channel -> receiver 0, second -> receiver 1, ...,
   up to `NUM_RECEIVERS=16`. Mirror of `Merge_res_New.py:253-256`.
2. **Amps-index translation:** in both `extract_signed_shapes_per_chain`
   (mask) and `build_export_payload` (decka loop), detect the
   pre-filtered case (`n_channels == len(response_channels)`) and treat
   amplitude index `k` as referring to measurement channel
   `response_channels[k]`. Untrimmed shapes (legacy / future code paths)
   continue to use identity. Sidecar JSON now includes
   `channel_to_receiver` and a populated `selected_r_indices` so the
   resolved mapping is observable.

### Validation

Offline run with the user's actual project data (275 chains from
PlyWoodTake1_4, mapping captured from `/modal/project_state`):

* Before: `decka_coeff non-zero: 0 / 2048` (the bug)
* After:  `decka_coeff non-zero: 896 / 2048 (43.8%)` = exactly
  128 modes x 7 response channels (perfect)
* Row 0 = `[0.0747, -0.3415, -0.2511, -0.7550, 0.6374, 1.0, 0.9761, 0, ..., 0]`
  matches the chain's raw `shape: [0.074713, -0.341522, -0.251068,
  -0.755036, 0.637383, 1.0, 0.976067]` exactly to 4 decimals.
* `sidecar.channel_to_receiver = {1: 0, 2: 1, 3: 2, 4: 3, 5: 4, 6: 5, 7: 6}`
* `sidecar.selected_r_indices = [0, 1, 2, 3, 4, 5, 6]`
