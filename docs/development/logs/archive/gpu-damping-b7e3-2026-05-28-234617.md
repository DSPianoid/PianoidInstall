## [PROGRESS 2 — 23:54] Damping path traced
**parameterKernel (Kernels.cu:80, 125):**
```cpp
int dump_coeff = pow((127 - *sustain_value) * dumper_position[stringNo], 0.6);
real dec_curr = coeff_gamma / (iterPerMs * 1000) + damper_string * dump_coeff;
```
- `dumper_position` is `dev_dec_open` (param name in kernel signature)
- `dec_open[s] = DUMP_CLOSED=1` → `dump_coeff = (127-sustain) * 1`^0.6
- `dec_open[s] = DUMP_OPEN=0` → `dump_coeff = 0` → no damper contribution
- Output: parameters[][3*arraySize+i] = `shift_b = (dec_curr - 1) * dec_inv` — this is what MainKernel reads as `t1` for FDTD.

**Live note-off path:**
1. NOTE_OFF → handleNoteOff → stageStringsForPitch(pitch, 0) → addStringToBatch(s, 0) → _add_string_for_playback(s, 0, 0) → dec_open[s] = DUMP_CLOSED
2. End cycle: dispatchBatch.commitStringBatch() → noStrings_in_GP=0 (velocity=0 didn't append) → new_notes_ind = 0+1 = 1
3. runSynthesisKernel: new_notes_ind=1 → upload dec_open + parameterKernel → addKernel (no gaussKernel because new_notes_ind == 1, not > 1)

This SHOULD work. Probe at synthesis.cu:203 confirms upload happens with correct values.

## Hypotheses ranked

### H1 (most likely): probe sees correct value but parameterKernel's read of `damper_string` is stale/zero on live path
- Live engine and audio thread might race against an in-flight preset swap.
- `dev_physical_parameters` is a TUNABLE field updated via `updateDerivedPointers` when a swap completes. If a parameter edit (e.g. damper_string POST) happens, working/updating buffers swap.
- BUT: `loadParameterToPianoid("dev_dec_open", ...)` writes to dev_dec_open's static-input dev_ptr. parameterKernel reads `physical_parameters[s+13]` (damper_string) from `dev_physical_parameters`. These are independent.
- However — IF the running preset was modified and damper_string got set to 0 for the strings in pitch 56, the formula `damper_string * dump_coeff = 0 * 19 = 0` → no damping regardless of dec_open.
- TEST: read back `dev_physical_parameters[201*16 + 13]` from GPU and check value.

### H2: parameterKernel writes new coefficients but addKernel reads OLD coefficients
- The dev_parameters buffer (output of parameterKernel) is read by addKernel via shift_0/shift_1/shift_b for each spatial point.
- Each point has its own coefficient slot. If parameterKernel correctly writes new shift_b for the damped note's stem and string points but the FDTD update reads them, it should damp.
- Check: there's no sync between parameterKernel and addKernel call (both are async on the same default stream → in-order on the stream). So sequencing should be fine.

### H3: dec_open isn't being uploaded for the RIGHT strings — pitch→string mapping
- `dec_open[201..203]=1` per probe. Does pitch 56 ACTUALLY map to strings 201..203?
- `getStringIndicesForPitch(56)` is what stageStringsForPitch consults.
- If for some reason the live preset has remapped pitch 56 to different strings (e.g. 220, 221), then upload-set [201..203]=1 affects strings the audio doesn't come from, and the actual sounding strings keep DUMP_OPEN=0.
- TEST: probe `getStringIndicesForPitch(56)` LIVE vs OFFLINE on same backend.

### H4: outerSoundChannel mismatch — damping applied but output picks up undamped feedback path
- MainKernel writes output for `pitch >= 128` strings (220+). The DAMPED string (201) is NOT the output string. Damping affects the feedin to modes. The modes feed back into the output strings.
- If string 201 gets damped (its FDTD coefficient changes shift_b), THEN feedin to modes drops fast. Modes then decay (their own internal damping). Output strings get less feedback.
- BUT — modes have their OWN decay coefficient (`mode_dec`). They may have a long decay even after string damping cuts the drive. This could be the slow ~937 RMS at 100ms.
- This is NOT a bug — it's correct physics. A piano damper on the string cuts excitation; modes continue to ring out at their own rate. The user expects the dampening to be IMMEDIATE in the output, but the modes are like the soundboard — they ring after the string is stopped.

## CRITICAL: H4 might be the answer
**This might not be a bug at all.** Per the docs (SYNTHESIS_ENGINE.md), output strings (pitches 128+) are soundboard proxies. Their displacement IS the summed mode feedback. The mode oscillators have their own damping coefficient. Damping the source string (e.g. pitch 56 → strings 201-203) stops *new* energy entering the modes, but the modes themselves continue to ring at their internal decay rate.

A 100ms RMS of 937 (vs initial RMS of much higher, presumably) might actually be CORRECT decay behavior for the soundboard — not for the source string.

## [FINAL — 00:15] Summary

**Code trace conclusion:** The live and offline paths use IDENTICAL code for note-off damping (same dispatcher, same runSynthesisKernel, same parameterKernel). No "wiring" difference exists between baseline fdf3dd2 and current 67148fa — only the `[dev-cbd5]` Pianoid.cu split (pure move-refactor), confirmed by git show of baseline body matching current.

**What parameterKernel does with dec_open:**
- Reads `dumper_position[stringNo]` (== `dev_dec_open[stringNo]`)
- Multiplies by `damper_string` (physical_parameters[s*16 + 13]) to scale the velocity-damping term `dec_curr`
- Writes `shift_b = (dec_curr - 1) * dec_inv` into the per-point coefficient buffer `dev_parameters`

**What MainKernel reads:** The pre-computed `shift_b` (`parameters[3*arraySize+i]`) which already encodes the damping.

**No derived state buffer exists for damping.** parameterKernel writes only `dev_parameters` (the per-point coefficient buffer). No mode/sound channel mirror. No string-side damping cache.

**Why the LIVE upload provably works:** `dev_dec_open` is STATIC_INPUT, single-allocation, single `dev_ptr`. Both `loadParameterToPianoid` and `getIntPointer` resolve to the SAME pointer. There's no double-buffer race possible on this buffer.

## Most likely remaining cause

Given the source code is provably identical and the upload provably reaches the kernel, my top hypotheses are:

**H1 (highest likelihood):** **The bug is NOT in the dec_open path at all.** Either:
- (a) parameterKernel is computing correct shift_b, but addKernel's per-point coefficient READ is stale (the `dev_parameters` buffer is per-point but maybe addKernel reads a different index). This needs a kernel-side probe to confirm.
- (b) The audio you're measuring (RMS ~937) is the **sound from the SOUNDBOARD modes**, not the source string. Modes have their own `mode_dec` decay coefficient. A piano damper stops the source-string excitation but the soundboard continues to ring at the modes' internal decay rate. **OFFLINE may damp faster because it doesn't process the same number of cycles after noteoff** — or the comparison RMS is taken at different time-after-noteoff.

**H2 (next):** **`damper_string` is actually zero or near-zero in the live preset for these strings.** Even though probe says dec_open=1 is uploaded, if `damper_string=0`, `dump_coeff` is multiplied by zero → no damping effect.
- Critical to verify: read back `dev_physical_parameters[201*16+13]` from GPU. If zero or near-zero, that's the bug — possibly a preset-load issue specific to LIVE path (a parameter edit during live playback that zeroed damper_string).

**H3 (lower):** **Pitch→string mapping diverges live vs offline.** Offline event queue might map pitch 56 → different strings than live. Probe dec_open at strings 0..255 and check which strings actually drive the source audio for pitch 56.
