## Ranked Hypotheses (Phase A end)

**H1 — "Mute is a destructive toggle on backend's `deck['feedback']` for the output pitch, and the GET-on-APPLY then re-displays the zero state". HIGH plausibility for the headline silence + cannot-restore-via-un-mute symptoms.**
- Mute click writes all-zeros into `Pitch(128+ch).deck['feedback']` on backend.
- APPLY click → `/load_preset` re-runs Pianoid initialization. If `read_from_preset` does NOT restore `deck['feedback']` for output pitches (because the preset JSON's `string_sound_channels` only restores `string_coefficients`, not `deck['feedback']`), then the engine's GPU still has zeros for that output pitch.
- But this would NOT silence ALL channels — only the one that was muted. The "ALL sound silenced" headline is unexplained by H1 alone unless the preset has 1 output channel OR H2 is in play.
- **Cannot-restore-via-un-mute**: between mute and un-mute, no APPLY happens. So `matrix` in React still holds the original values. Un-mute click → `muteMap = all-1`, `mutedRow = matrix * 1 = original values` → POST original values → backend restores `deck['feedback']`. **Should restore audio.** UNLESS: the un-mute does NOT re-emit (e.g., emit short-circuits when `change.zone === "Cell"` and pitch hasn't changed), OR the un-mute gesture is not the same as the mute gesture.
- **Survives engine reload** = APPLY click. After APPLY: presetVersion bumps → init(GET response). The GET fetches the NOW-zero `feedback`/`output` from the backend's `pitch.deck['feedback']`. `init` resets muteMap to all-1, matrix to all-zero. Display shows "all unmuted, all zero". User clicks a cell to un-mute — but muteMap was already all-1 (so click toggles to mute), and the matrix being zero means any subsequent edit can't restore the original values without manual entry.
- This explanation is the "structural leakage from UI": the **frontend's transient mute state is being aliased onto the backend's PERSISTENT physical-parameter store** (`deck['feedback']`), without a separate engine-side concept of "mute".

**H2 — "Single-channel mute is in fact a multi-key emit". MEDIUM plausibility for the headline.**
- If `change.zone` somehow becomes "pitchesVector" instead of "modesVector" (e.g., the matrix-orientation toggle `matrixRowIsPiano` interacts oddly), `affectedKeysFor` returns ALL keys → POST for every output pitch → silences all channels.
- Or `applyImperativeMuteToggle` (dead in normal path) gets triggered somewhere and walks ALL keys.
- Easy to discriminate by network capture in Phase B.

**H3 — "Save-on-edit auto-persists muted state to preset on disk". MEDIUM plausibility for the cross-reload survival.**
- Need to check if the frontend or backend auto-saves after a /set_parameter/feedback. If so, a true backend restart would still read the muted preset.

**H4 — "The engine-side aggregate doesn't notice when only 1 of 4 channels carries the bulk of the signal; muting any 1 channel silences perceived audio". LOW plausibility — physics-level.**
- For a multi-channel preset where one channel happens to carry the dominant signal energy.

**H5 — "applyImperativeMuteToggle wired to onMuteMapChange has a different code path that emits all-rows; it's silently invoked from a code path I haven't found". LOW-MEDIUM plausibility.**

(a) **NOT clicking APPLY after backend restart** — sound stays silent because no engine. (Trivial, not a bug.)
(b) **Clicking APPLY after backend restart, expecting sound to come back** — but it doesn't. This is the real bug. Frontend's APPLY does `setSoundChannelData(null) → POST /load_preset → GETs → bump presetVersion`. The GET should fetch the preset's clean values from disk. Init should reset React state from those clean values. **So why does silence persist?**

**H1' — Init useEffect double-fire reseeds from intermediate state.** Two init() calls happen on APPLY due to sequential awaits — one when soundChannelFeedbackMatrix changes, one when presetVersion bumps. If the React render between sets the matrix value to clean data but the muteMap reset is lost in a stale closure, mutedRow at next emit could be all-zero. **Discriminator (Phase B):** instrument `init()` and count how many times it fires per APPLY; verify React state of matrix + muteMap immediately after APPLY = (fresh data, all-1). If matrix is clean but next user click still POSTs zeros, this is in play.

**H2' — POST payload axis-flip or full-matrix POST.** The single-cell edit POST might be carrying a transposed matrix or sending more than just the one cell's row. **Discriminator:** chrome-devtools network capture of a SINGLE click. Inspect URL (single-pitch endpoint vs `/all`/`output`), payload shape (one-key vs many-key dict), and per-row length (matches num_modes vs num_pitches).

**H3' — Backend `feedback` handler side-effects spread to other pitches.** Maybe the `update_deck` call has a side effect on neighbouring pitches' decks, OR the same handler ALSO calls `update_sound_channel_coefficients` which mutates `string_coefficients` for ALL pitches. **Discriminator:** REST replay — `curl POST /set_parameter/feedback/128` with a small payload, then `GET /get_parameter/feedback/output` and `GET /get_parameter/string_sound_channel/all`. Compare to pre-POST state. If MORE than the addressed pitch changed → H3'.

**H4' — Fresh APPLY after a backend restart leaves the engine in a broken initialization state.** Maybe `/load_preset` on a backend that was just restarted (vs cold-started by the launcher) leaves the pianoid object in a half-initialized state — driver opens but parameters never push. **Discriminator:** after backend stop+restart+APPLY, capture `/health` AND directly inspect engine state by sending a `/play_note` and checking `/get_chart_test?chart=note_playback`. If health says `pianoid_loaded=true` but offline render is silent → H4'.

**H5' — Frontend's `presetVersion`-driven init() races with a debounced POST in-flight from before the APPLY.** A debounced `changeSoundChannelFeedback` scheduled BEFORE the user clicked APPLY (50-300ms window) fires AFTER the GETs landed but BEFORE the init() consumed the GET data, re-corrupting `soundChannelFeedbackMatrix` to the muted state. `cancelPendingParamWrites` is only called in `switchPreset`, NOT in `loadPreset` — that's an asymmetry. **Discriminator:** chrome-devtools timeline of an APPLY immediately after an edit, with the debounced write window visible. Look for a `/set_parameter/feedback/<pitch>` POST that fires AFTER the `/load_preset` POST settled.

**H6' — Frontend state corruption in `useMatrixHistory` is destructive to the matrix on edit (not just muteMap).** A bug where `applyChange` with `operation:"Value"` accidentally fills the WHOLE row with the input value (or with zeros) for axis ordering reasons. The `zone:"Cell"` branch at useMatrixHistory.js:116-120 does `newMatrix[change.pitch][change.mode] = change.newValue` — single cell — fine. But the `zone:"modesVector"` branch at line 121-125 does `newMatrix[change.pitch].fill(change.newValue)` — fills the WHOLE row with one value. If a Cell click is misclassified as modesVector, an edit to one cell with value=200 would silence everything in that row (and POST that row's array). **Discriminator:** the same chrome-devtools click capture. If a Cell click shows up as `zone:"modesVector"` upstream, H6' wins.

**H7' — Bootstrap on fresh npm DOES use localStorage for the matrix somehow (refutes "React state only").** I confirmed there's NO localStorage key for the matrix, but maybe there's a stealth cache (sessionStorage? IndexedDB? React Query?). **Discriminator:** chrome-devtools — dump `localStorage`, `sessionStorage`, and inspect for any IndexedDB databases. If npm restart works but it's because of localStorage → H7'. (LOW plausibility — full grep confirmed nothing.)
