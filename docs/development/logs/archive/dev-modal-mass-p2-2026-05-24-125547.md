### Phase 1 H1 unit audit — 2026-05-24T13:30:30Z

**Audit target:** `compute_frf_for_cycle_stacks` (frf_orchestrator.py:227-328), what units does the persisted H1 carry?

**Pipeline trace:**

1. `_extract_scenario_cycle_stacks` (frf_orchestrator.py:379-566) — loads `raw_recordings/raw_*_chN.npy` (numpy arrays of raw ADC samples — voltages). Skips `normalize_by_calibration` per Q1=B (line 528-538). Cycle alignment is applied (time-domain shift only, no amplitude scaling). Result: `force_cycles` and `response_cycles` are **raw ADC voltage time series, NO calibration to physical units**.

2. `compute_frf_for_cycle_stacks` (line 271-328):
   - Force is windowed (5 ms around impact peak) — preserves voltage units; the window mask is dimensionless 0/1 with a 0.2 ms half-cosine fade (also dimensionless).
   - `X = rfft(windowed)` — complex spectrum of force voltage; units `volt·sample` (or equivalently `volt/Hz`-scaled).
   - `Y = rfft(response)` — complex spectrum of response voltage; same units `volt·sample`.
   - `Sxy = Y · conj(X)` — units `(volt·sample)²` = `V²·s²`.
   - `Sxx = X · conj(X).real` — units `V²·s²`.
   - `H1 = mean(Sxy) / mean(Sxx)` — units cancel to **dimensionless** (ratio of two voltage spectra).

**Conclusion: H1 is in `response_voltage / force_voltage` — a dimensionless ratio, NOT physical receptance (m/N) and NOT a normalised-to-peak ratio.** Both numerator and denominator are RAW ADC voltages with no calibration applied. The sensor transfer functions (force-sensor sensitivity V/N, response-sensor sensitivity V/m or V/(m/s²)) and the ADC scaling factors are folded into H1 as a multiplicative constant per scenario (or per channel).

**Implication for shape_inversion:**

Since H1 (and hence residues `R_n = ω_n² · ζ · H_peak · 2j` etc) carry an unknown but constant scaling factor per (response_channel, force_channel) pair across all scenarios, the recovered `m_n` is in "internally-consistent voltage-ratio units" — NOT kg. The **ratios** `m_n / m_ref` are physically meaningful (the scaling factor cancels in the ratio) ONLY when:

- The same response channel is used for both modes (otherwise different sensor sensitivities apply), AND
- The same force channel is used for both modes (same force-sensor sensitivity, same hammer dynamics).

Since we use ONE force channel (one hammer) and pick the dominant response channel per chain (the chain's strongest receiver), the scaling factor IS constant across modes for a given chain set and the relative-mass extraction is well-posed.

**Decision (per fix plan branch 2):** Even though H1 is non-physical, we adopt the **dual-unit-max convention** as planned because:

- It's internally consistent across modes (key requirement).
- It matches the textbook EMA convention (max|φ| = 1 per mode) which we already document as the shape-normalisation choice.
- The recovered `m_n` is in "internally-consistent units"; `m_relative = m_n / m_ref` is the user-meaningful output and is the same regardless of whether H1 is physical or scaled.

The orchestrator docstring + REST endpoint payload must document the units as "internally-consistent dimensionless modal-mass ratio" (NOT kg) per user Q2 = Relative-only. Absolute SI calibration is deferred per the proposal.

**No doc changes needed for H1 itself** — `frf_orchestrator.py` lines 60-93 already state H1 is the dimensionless ratio without claiming physical units. The shape_inversion docstring + the new orchestrator docstring will state the relative-units convention explicitly.

[DATA-MODEL-FACT] H1 = response_voltage / force_voltage, dimensionless, raw ADC. Relative modal mass m_n/m_ref is well-posed; absolute kg requires future user-side calibration (proposal §5.5.1).

### Kernel rewrite: shape_inversion.py dual-unit-max convention — 2026-05-24T13:35:00Z

**Bug:** Original kernel computed `m_n = 1/max|response_shape|` where `response_shape = sigma0 * max|u0| * v0` (raw SVD right singular vector). That makes m_n depend on the random extremes of `v0`, which vary per mode independently of the true modal mass. 2/14 tests failed with ~30% mass error.

**Fix:** Adopt dual-unit-max convention. Both actuator AND response shapes normalised to max-abs = 1 per mode. Then the residue at the joint peak equals 1/m_n by the physics; equivalently `m_n = 1 / (σ_0 · max|u_0| · max|v_0|)` where σ_0 is the leading singular value of the rank-1 SVD of `R_n`.

**Algorithm (mathematically):**
1. SVD of usable rows of `R_n` (shape A_u × S): `R_n ≈ σ_0 u_0 v_0^H`
2. Pre-normalise: `M_a = max|u_0|`, `M_v = max|v_0|`; `actuator_shape = u_0 / M_a`, `response_shape = conj(v_0) / M_v` (Vh row → response convention)
3. `m_n = 1 / (σ_0 · M_a · M_v)` — directly from the outer-product peak identity
4. Phase-align: rotate so actuator peak is real-positive; response absorbs conjugate phase (preserves residue)

**Test fixture rewrite:** both `shape_actuator_true` AND `shape_response_true` now unit-max-normalised per mode in `synthetic_3_mode_known` fixture + the missing-actuator test. Under this convention masses_true=[1, 2, 4] is consistent with what the kernel recovers.

**rfp_fit.py cleanup:** removed dead code (first `lstsq(A_real, b_real)` was overwritten by the block-matrix solve 6 lines later; the naive vstack formulation forces imag(x)=0 which is wrong for complex residues). 4/4 RFP tests still pass.
