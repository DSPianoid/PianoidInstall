### Phase B M2 — design refinement BEFORE implementing — 2026-05-31T04:50:00Z

While inspecting the existing float-ring writer (Pianoid_synthesis.cu:609 appendCycleAudioToHostBuffer + Pianoid.cu:142 cycle_parameters[3] = mode_iteration), discovered:
- `cycle_parameters[3]` (samplesInCycle) is initialised to `mode_iteration` at construction.
- The production float ring uses `chunkSize = mode_iteration * num_channels` — i.e. THE SAME extent as my proposal's per-channel-samplesInCycle approach when at default config (which is always — setCycleIterations is rarely used).
- Why the float ring works while dev-soundint-live's Sint ring railed: float-typed uninitialised GPU memory reads as ~0; Sint32-typed uninitialised memory reads as ±INT_MAX. The float ring's "uninitialised tail" never surfaced because float-0 is benign.

★ DESIGN REVISION: mirror the float-ring extent verbatim (`mode_iteration * num_channels`) for the Sint ring rather than the per-channel-samplesInCycle pattern.
  - Pro: tested-correct production extent; same per-cycle chunk size as float ring; cleaner code.
  - Risk: if `setCycleIterations()` is ever called with a value < mode_iteration, the per-channel tail of the Sint ring will read as ±INT_MAX garbage (same defect dev-soundint-live measured). Mitigation: add a per-cycle cudaMemset of dev_soundInt FIR-OFF path (the FIR-ON path already memsets at line 553). This costs ~mode_iteration*num_channels*4 bytes per cycle but eliminates the failure mode entirely.

This is the simpler, more correct fix the user's "make multi-channel native" directive points at — the float-ring extent is the canonical contract. The proposal's per-channel-samplesInCycle approach was an overcautious response to a misdiagnosed bug.

The post-FIR float ring is unaffected (FIR ON path always memsets dev_filteredSoundFloat at line 447).
