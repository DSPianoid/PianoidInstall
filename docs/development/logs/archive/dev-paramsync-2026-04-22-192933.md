**Answers to the four questions the orchestrator asked:**

1. **UI param change → GPU write path** — two classes:
   - **TUNABLE preset params** (physics, hammer, excitation, modes, deck, volume_coeff): REST/WS → `pianoid.update_parameter()` → `ParameterManager.*` → C++ `updateTunableParameter(name, data)` → async double-buffer pipeline (IDLE→UPDATING→SWAPPING→SYNCING→IDLE) on a dedicated `update_stream_`. Pointer swap `dev_preset_working_ ↔ dev_preset_updating_`. Kernels in flight keep reading old working copy (docs claim atomic swap).
   - **RUNTIME params** (volume_level, deck_feedback_coefficient): REST/MIDI CC → `pianoid.set_volume_level()` / `set_deck_feedback_coefficient()` → C++ `setRuntimeParameters()` → **direct `cudaMemcpy` to `dev_main_volume_coeff`** (docs: "no double-buffer swap needed; scalar via atomic or direct cudaMemcpy"). No cycle-boundary gating documented.

2. **Thread that writes vs thread that runs kernel:**
   - Writes: Flask HTTP thread (and MIDI listener thread for runtime CC), protected by Python `self.cuda_lock` (a `threading.Lock`).
   - Kernel: engine thread `OnlinePlaybackEngine.run()` launches `addKernel` each cycle via `pianoid_->runCycle({Online, ...})`.
   - `cuda_lock` only serialises Python-side callers; it does NOT gate the engine thread's kernel launch.

3. **Current synchronisation mechanism:**
   - Tunable params: async double-buffer + `update_complete_event_` / `sync_complete_event_` + background **poll thread** (`updatePollThread` in UnifiedGpuMemoryManager) that advances the state machine.
   - Runtime params: **none — direct write to a live GPU buffer** that `addKernel` reads.
   - `DROP_IF_BUSY` policy on tunable: overlapping tunable updates are dropped (hence `ParameterManager._gpu_upload` calls `waitForParameterUpdate()` to avoid silent drops).

4. **Poll thread (orchestrator's "100μs poll thread"):** `updatePollThread` in `UnifiedGpuMemoryManager`. Docs describe its role (state machine advancement on event completion) but don't document the sleep interval — code reading required to confirm "100μs". This is the scheduler for tunable state transitions; it is NOT gated to cycle boundaries.

**Key finding:** The existing double-buffer gives us an almost-right primitive — kernels read from a pointer that is only swapped between kernel launches. But:
- The swap is gated by CUDA event completion, not by the engine thread's cycle cadence. A pointer swap could in principle occur mid-kernel-launch (the doc claim of atomicity depends on this being rare).
- Runtime params (volume, feedback_coefficient) completely bypass this mechanism.
- There is no explicit "apply queued param updates at cycle start" hook in `OnlinePlaybackEngine.run()` or `Pianoid::runCycle`.

**Baseline evidence from orchestrator:** iter=4 near-clean, iter=8 distorts. Longer cycles widen the window in which a runtime-param write can land during kernel execution. Consistent with the hypothesis that the interference is real and cycle-length-sensitive.

**Concurrency guarantees (actual):**
- `cuda_lock` (Python `threading.Lock`) serialises Python-side writers only.
- `update_mutex_` (C++ `std::mutex`) protects `updateTunableParameter` re-entry and the host-side `swapBuffers` bookkeeping. It is NOT held by the engine thread during the cycle.
- Kernel launches happen on the default NULL stream (not a dedicated synth stream — F5 has only moved the audio `produce()` off default). A sync `cudaMemcpy` on the default stream therefore inserts an implicit sync point relative to the kernel.

**Double-buffer primitive — how much does it already give us?**
- Tunable param writes never land on the live `dev_preset_working_` directly. They land on `dev_preset_updating_`.
- The pointer swap happens on the host from a background poll thread with a 100μs poll period. The swap is protected by `update_mutex_` only; it does not coordinate with the engine thread's kernel launches.
- In practice kernels capture the pointer at launch and hold it for the duration of their execution. The pointer swap changes only the host-side variable used for the **next** launch. So tunable params *effectively* take effect at a cycle boundary — but it's an emergent property, not an enforced contract.
- `syncBuffers()` D2D copy on `update_stream_` is concurrent with the kernel on the default stream. CUDA allows this (different streams, different source/dest within source bounds). The D2D reads from `dev_preset_working_` — the now-live buffer the kernel is also reading — which is safe (two readers, no writer) but the D2D's source bytes must not be mutated meanwhile. They aren't (the kernel only reads them). OK.

**Existing infrastructure we can leverage:**
- `PARAM_UPDATE_SINGLE` / `PARAM_UPDATE_BATCH` event types already exist (`PlaybackEvent.h`, `EventDispatcher.cu:106`). Events are drained at cycle start inside `OnlinePlaybackEngine::run()` via `processEventsAtCycle(cycle_index)` — this is **the exact "between cycles" hook** the orchestrator wants.
- `RealTimeEventBuffer` is a thread-safe producer-side channel: any REST/WS/MIDI-CC thread can push an event, and the engine drains it at cycle start.
- `processEventsAtCycle` is called **before** `pianoid_->runCycle()` — i.e. before the kernel launch — so anything applied in its handlers executes on the engine thread, outside kernel execution.

**Evidence of interference paths today:**
1. `set_volume_level` → sync `cudaMemcpy` — WILL interfere with a live kernel (forces host<->device sync on default stream).
2. `set_deck_feedback_coefficient` — same pattern.
3. Tunable params — in principle safe, but depend on the poll thread racing the engine thread favourably. Under bursty update traffic with iter=8 cycles (~85μs per iter × 8 = ~680μs GPU kernel time) and a 100μs poll, the poll can advance state during the kernel, land a `cudaEventRecord` on `update_stream_` while the kernel is still running on the default stream, and then the subsequent sync `cudaMemcpy` (or the D2D in `syncBuffers`) can introduce scheduling perturbation.
4. Any direct `cudaMemcpy` during kernel execution (default-stream sync memcpy) serialises against the kernel.

Step 2 conclusion: the codebase already contains two of the three pieces needed:
- Per-cycle event drain hook (`processEventsAtCycle`).
- Thread-safe producer channel (`RealTimeEventBuffer`).
What's missing: a uniform discipline that **all** host → GPU parameter writes go via this hook, and a contract that the hook is the **only** place such writes can happen.

### Step 3: Design doc — 17:05

- Branch created: `feature/cycle-synchronized-param-updates` (PianoidCore, from `dev` @ `bbf2453`).
- Design doc written: `docs/development/PARAM_SYNC_DESIGN.md`.
- Staged in repo-root git (not yet committed — awaiting user approval per orchestrator directive).

**Design summary (one paragraph):** Enforce invariant that all host→GPU parameter writes happen on the engine thread inside `processEventsAtCycle`, strictly between kernel launches. Leverage existing `RealTimeEventBuffer` + `PARAM_UPDATE_*` event types to enqueue from producers (Flask / WS / MIDI) and drain from engine. Add `PARAM_UPDATE_RUNTIME` (scalars) and `PARAM_UPDATE_BULK` (tunable params) + `PARAM_UPDATE_PRESET_SWITCH` payload variants. Retire the 100μs poll thread; replace with `memory_manager_.serviceAsyncUpdates()` called at cycle start on the engine thread. Insert `cudaStreamWaitEvent` on the kernel stream so param memcpys on `update_stream_` complete before kernel launch — structural guarantee of ordering. Producers become enqueue-and-return. `setRuntimeParameters`, `updateTunableParameter`, and `switchPreset` gain a precondition check (off-engine-thread calls during playback → log error + return false); direct calls still allowed at startup / offline / semi-offline where no engine thread is running.

**Testability (quantitative, not subjective):** Add debug-build counter `concurrent_writes_during_kernel_` that increments if any `cudaMemcpy*` fires from a non-engine thread while `isApplicationRunning()`. Test battery: 30s active playback under 100 param-edits/sec + 30s MIDI CC volume at 1kHz + preset switch every 500ms — counter target = 0. Second assertion: bit-identical offline WAV across two runs of the same MIDI + param-edit script (proves cycle-deterministic application).

**Risk budget:** +5μs unconditional per cycle + ~5μs per param event handled. Real-time budget at iter=8 has ~650μs headroom. Well within the orchestrator's 100μs budget.

**Open questions flagged to orchestrator:** (1) dedicated synth stream coordination with F5 agent; (2) keep or drop Python `cuda_lock`; (3) scope of `setInitializationParameters`; (4) counter-first vs producers-first implementation order.
