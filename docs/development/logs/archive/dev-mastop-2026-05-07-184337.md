## Key design decisions

- **Pre-flight runs on EVERY preset load**, not just when starting a fresh backend: every `/load_preset` re-opens the Pianoid audio driver (load_preset always destroys+recreates the engine per OVERVIEW.md:187), so the MA-stop guard must run unconditionally when modalRunning is true.
- **Audio-driver close confirmation = process death:** the launcher's existing `gracefulShutdown` waits up to 3 s for the MA process to exit (via `process.kill(pid, 0)` polling) and falls back to `taskkill /T /F`. Once the response returns, the OS has reclaimed any audio-device handle MA was holding. No additional polling-vs-event design was needed — process death IS the signal.
- **Backend `/shutdown` strengthened to cancel + join MeasurementSession before SIGTERM:** so audio-driver release is deterministic instead of relying on OS-level process death cleanup, which was racing the next `/load_preset` call.

## Investigation findings

- **Current preset-load flow (PianoidTuner.js:316-412):** probe `/health`, kill stale if launcher doesn't own it, start backend if needed, `loadPreset()`. No MA awareness whatsoever before this fix.
- **Audio-driver coupling:** Pianoid main owns the SDL3/SDL2/ASIO device whenever the engine is initialised. MA owns the device only DURING `MeasurementSession._invoke_collection` (lines 310-354 of collection_engine.py). The reverse-direction guard (`MeasurementSession._pause_backend` → `POST /pause_synthesis` on Pianoid) was already in place; the forward direction (Pianoid load → stop MA) was missing.
- **Modal adapter `/shutdown` BEFORE this fix:** spawned a 0.3 s sleep + SIGTERM. Did not cancel any in-flight session. Audio device handle was reaped non-deterministically by OS process death.
