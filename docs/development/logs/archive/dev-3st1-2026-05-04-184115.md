**KEY FINDING — `_merge_split_chains` IS NOT CALLED IN THE SLIDING-WINDOW PATH.**

Trace:
- `EspritRunner.run_tracking()` → `_run_tracking_sliding_window()` (when `tracking_method="sliding_window"`, the default).
- `_run_tracking_sliding_window()` calls `track_modes_along_bridge(all_detections, config=config)` (L356).
- `track_modes_along_bridge()` dispatcher routes to `_track_sliding_window(converted, config)` (L753-754) and **returns immediately at L754** — never reaches the sequential post-processing block (L890-904) where `_merge_split_chains` is called.
- `_track_sliding_window()` itself ends at L695 with `return chains` — no merge step.
- The sequential post-processing path (`_merge_split_chains`, sort, renumber, trend trimming at L890-904) is dead-code for any sliding_window invocation.

**This is the root cause of the user's failure case.** The sliding window uses **greedy peeling** — once a confirmed cluster is created at frequency window centred at e.g. 440 Hz (covering scenarios 0-40), its detections are removed from the unassigned pool. When the window slides to 445 Hz (covering scenarios 50-90 of the SAME physical mode that drifted by 5 Hz across the bridge), those detections form a SECOND independent cluster.  Both pass `sw_min_cluster_scenarios=3`. Both have HIGH per-bridge coverage. They are NON-OVERLAPPING in scenario coverage. Their reference shapes are SIMILAR (same mode → MAC>0.5). They never get merged because:
1. `_track_sliding_window` doesn't call `_merge_split_chains`.
2. Even if invoked, `_merge_split_chains` uses `freq_tol_pct * 2 = 0.06` for the chain-mean proximity gate — for 440 vs 445 Hz that's `5/440 = 0.011 < 0.06`, so they WOULD pass the freq gate. The junction freq diff `(445-440)/440 = 0.011 < freq_tol_pct * 3 = 0.09` — passes. MAC > 0.5 — passes. So this case would in fact merge correctly under `_merge_split_chains` if it were ever called.

**Implication for planning report:**
- **Option A** (cheap fix): wire `_merge_split_chains` (with grid-aware skip already in place) into the sliding-window path. ~15 LOC change. Could resolve the user's failure case without writing the new algorithm.
- **Option B** (build new algorithm anyway): user's spec is more principled, supports the full coverage×overlap matrix, and stray-point assignment recovers detections that are currently dropped.

**Strong recommendation:** do BOTH in this PR.
1. **First**, the cheap A-fix wired into the sliding-window path (and lock it behind the same `tracking_method="sliding_window"` so existing tests remain stable). This is a 15-LOC bug fix that may already address the user's specific complaint.
2. **Then**, the full Stage-3 nuclei-merge algorithm as a new `tracking_method="nuclei_merge"` option in a new sibling module `mode_tracking_nuclei.py`. This avoids growing `mode_tracking.py` (already RED at 1277 LOC, +24 since dev-c969 sync) and keeps the new code reviewable in isolation.
