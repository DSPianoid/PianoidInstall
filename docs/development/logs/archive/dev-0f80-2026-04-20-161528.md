Race window exists for both `loadPreset` and `switchPreset` paths.
- **Options considered:**
  - **Option A (ack/reconcile):** Change backend `/load_preset` and `/preset/switch` responses to include SC snapshot, and have frontend replace state atomically on ack. Requires backend changes + cross-repo coordination. Heavy and crosses module boundaries.
  - **Option B (clear cache on preset switch):** `setSoundChannelData(null)` + `setSoundChannelFeedbackMatrix(null)` BEFORE the fetch in both `loadPreset` and `switchPreset`. `getSoundChannelData` already replaces (not merges) so post-clear fetch gives clean state. Minimal (4 lines), single-file, respects P1 authority.
  - **Option C (useSoundChannels-side coordination):** Would invert the flow — the hook should consume state, not invalidate it.
- **Decision: Option B**
  - Rationale: P1 — `usePreset` owns `soundChannelData`, keep it the sole writer. P2 — preset transition is destructive; merge semantics are wrong for it. Minimal change, no backend surface change, no new contract.
  - Also kill any pending debounced SC writes on preset change (they carry stale data). Current debouncedSetSoundChannelRef has no cancel hook — easiest: clear state synchronously, and debounced write from pre-switch will complete harmlessly (but backend will accept and immediately be overwritten by fresh state).
  - Actually a better pattern: null-out state → next render, `useSoundChannels` sees null and bails (line 250). Pending debounced HTTP writes to backend are harmless because the backend rebuilds state from the new preset anyway (via `/load_preset` which destroys the pianoid instance, or `/preset/switch` double-buffer swap). The debounced HTTP writes land in the VOID — if the target pitch/channel doesn't exist in the new preset, backend rejects with 416; if it does exist, that write hits a pitch/channel with meaning in new preset, which is a pre-existing bug and not the scope of this task.
