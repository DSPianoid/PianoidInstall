## Goal

Conduct a 4D grid search for the **Ultra-Low band** ESPRIT parameters
(`skip_start_ms` = "cut in", `ir_length_ms` = "cut out", `start_fade_ms` = "fade in",
`end_fade_ms` = "fade out") to find the configuration that maximises the **Stage-1
nuclei-detection quality metric**. Run only Stage 1 of the `nuclei_merge` tracking
algorithm (no Stage 2 merge / Stage 3 stray assignment).

**Two-phase strategy:**

- **Phase A** — Grid search at `decimation=16` (vs default 4) for ~16× faster ESPRIT.
  Find top-K configs by quality metric.
- **Phase B** — Re-run the top-K Phase-A configs at `decimation=4` (and optionally 2)
  to verify the optimum holds at native resolution. Discuss if the optimum shifts.

## High-stakes data-model facts (CLAUDE.md)

- **Ultra-Low band defaults (EXTENDED_BANDS[0]):** `f_min=30, f_max=100, filter_order=4,
  decimation=4, exp_factor=0.15, model_order=8, ir_length_ms=1000, skip_start_ms=50,
  start_fade_ms=10, end_fade_ms=20`.
- **Sample rate** of averaged signals: **fs = 48000 Hz** (verified from PlyWoodTake1_4
  metadata). Decimation 4 → fs_band = 12000; decimation 16 → fs_band = 3000 — STILL
  >> 2 × f_max (100 Hz) so anti-alias margin is preserved (Nyquist = 1500 Hz at
  dec=16).
- **Averaged-response duration in available datasets:** **300 ms (14400 samples at
  48 kHz)** — measured on `D:/modal_measurements/PlyWoodTake1_4`. So `ir_length_ms`
  cannot exceed 300; `band_processing.process_band` silently clamps to available
  samples. Effective grid for `ir_length_ms` ∈ [100, 300] ms.
- **Stage-1 inputs:** `_detect_nuclei` takes a flat `List[ModeDetection]` and
  `total_scenarios`. The `config.nm_nucleus_*` knobs (window_pct=0.03, min_width=3.0,
  mac_threshold=0.7, min_detections=4, min_scenarios=4) are held at defaults.
- **Channel filter:** `response_channels = [1..7]` (channel 0 = calibration). Already
  excluded by `mapping.response_channels` in `run_all_points`; for the grid harness
  we slice `signals[:, 1:]` directly since we don't load mappings.

## Quality metric — proposed (PAUSE FOR ORCHESTRATOR/USER APPROVAL)

For Stage-1-only nuclei detection, the goal is "find as many cohesive single-physical-mode
clusters as possible without admitting junk". I propose a composite score:

```
Q = N_nuclei_qualified * mean_cohesion * mean_coverage_rate

where:
  - N_nuclei_qualified = count of Stage-1 nuclei that pass a stability filter
    (>= 4 detections AND >= 4 distinct scenarios, which is already the Stage-1
    hard gate; we additionally require coverage >= 0.5 to reject weak nuclei).
  - mean_cohesion = mean across nuclei of intra-nucleus reference-shape MAC, computed
    as the mean pairwise MAC between every detection's mode_shape and the nucleus
    reference shape. Range [0, 1] with HIGHER = TIGHTER cluster.
  - mean_coverage_rate = mean across nuclei of (n_detections / total_scenarios).
    Range [0, 1] with HIGHER = nucleus spans more bridge.
```

**Rationale:**
- Multiplicative: a config that finds 20 weak nuclei (coverage 0.2 each) should NOT
  beat a config that finds 8 strong nuclei (coverage 0.8 each). The multiplicative
  form penalises the "spam-many-weak-clusters" failure mode.
- Cohesion is the Stage-1 promise — nuclei MUST be tight. We enforce this via the
  intra-nucleus MAC (NOT the inter-nucleus MAC that Stage-2 uses).
- Coverage rewards configs that produce nuclei spanning more of the bridge — the
  ultimate goal of tracking.

Secondary metrics also recorded for diagnostics:
- `n_nuclei_total` — raw nucleus count (no coverage filter).
- `n_unassigned` — detections that did NOT join any nucleus (Stage-3 candidates).
- `mean_freq_drift_pct` — mean per-nucleus (max-min)/mean of frequencies. Lower =
  tighter freq cluster.
- `total_runtime_s` — Phase-A speed reference.

## Parameter grid — proposed (PAUSE FOR APPROVAL)

| Parameter | Range | Step | Levels | Note |
|-----------|-------|------|--------|------|
| `skip_start_ms` (cut in)   | 0 – 100 | 25 | 5  (0,25,50,75,100) | covers default 50 ± 50 |
| `ir_length_ms`  (cut out)  | 150 – 300 | 50 | 4  (150,200,250,300) | bounded by 300-ms available data |
| `start_fade_ms` (fade in)  | 0 – 30 | 10 | 4 (0,10,20,30) | covers default 10 ± 20 |
| `end_fade_ms`   (fade out) | 0 – 40 | 10 | 5 (0,10,20,30,40) | covers default 20 ± 20 |

**Total grid points: 5 × 4 × 4 × 5 = 400.**

Cost estimate: PlyWoodTake1_4 has 30 scenarios. Each grid point = 30 ESPRIT runs (one
per scenario, ULTRA-LOW band ONLY — bypass `merge_multiband_results`, call
`process_band` + `esprit_modal_identification` directly). At decimation=16 each ESPRIT
is small (model_order=8, ~100 samples after 300ms × 3000Hz × skip — Hankel size ~50).
Estimate ~0.05 s per scenario at dec=16 → ~1.5 s/grid point → ~10 min total Phase A.
At dec=4 (Phase B): ~0.2 s/scenario → ~6 s/grid point. Phase B re-runs only top 5 → ~30 s.

## Dataset choice — proposed (PAUSE FOR APPROVAL)

**`D:/modal_measurements/PlyWoodTake1_4`** — 30 scenarios, 8 channels (ch0 calibration
excluded → ch1–7 used as response), 48 kHz, 300 ms averaged signals. Most-recent take
in the `PlyWoodTake1_*` family (the `_4` suffix is the latest revision the user has
been working on). All scenarios have `averaged_responses/average_ch{0..7}.npy`.

Bridge-aware semantics (bass/treble split) are NOT relevant for Stage 1 — the nucleus
detector is layout-agnostic (sorts detections by frequency, doesn't reference scenario
ordering). Bridge boundary unused.

## Decimation Phase B target

- **Phase A:** decimation=16 (sweep all 400 grid points).
- **Phase B:** decimation=4 (the EXTENDED_BANDS default) on top-5 configs from Phase A,
  AND on the `decimation=16` Phase-A optimum (so we have a direct same-config
  decimation comparison).
- **Optional Phase B' (only if user approves runtime):** decimation=2 on the top-1
  config — checks whether further decimation reduction helps.
