## Phase 3 — Top suspects with hypotheses

### Suspect S1 (LOW direct, MEDIUM indirect): dev-bfe2 preset working-copy model

Commits: `d3958d8`, `ca014f3`, `1d30856`, `c2d7c21`, `743c512`.

**What changed**:
- `_load_preset_to_library` is now invoked TWICE on init — once for the original (under the preset's file name `Belarus_8band_196modes-MFeq`) and once for the auto-spawned working copy (e.g. `Belarus_8band_196modes-MFeq (working 1)`). The user is ACTIVE on the working copy, not on the original.
- `switch_preset` `saveActiveToLibrary` is gated on the active being editable.
- `_assert_active_editable` raises on any param-edit facade for read-only entries.
- `switch_preset` now `waitForParameterUpdate()` before `saveActiveToLibrary` (added in c2d7c21 to defuse DROP_IF_BUSY races).
- `send_deck_params_to_CUDA()` is called inside switch_preset.

**Could it explain live-only damping failure?** Not directly — the working copy GPU slot is loaded via the same `_load_preset_to_library` → C++ `loadPresetToLibrary` path that writes `damper_string` into `dev_physical_parameters`. The DAMPER_PROBE already confirms 3.6e-05 live. If the working slot were wrong, the offline render (which reads the same `dev_physical_parameters` of the active slot) would also damp poorly — but offline damps brilliantly. So dev-bfe2 alone is unlikely to cause "live damps but offline doesn't".

**Where it could indirectly matter**: the explicit `waitForParameterUpdate()` + the conditional `saveActiveToLibrary` + the additional `send_deck_params_to_CUDA()` introduce new async / double-buffer interactions. If an in-flight async upload from the original slot's snapshot intermittently overwrites the working slot's `dev_dec_open` or `dev_parameters` shift_b under DROP_IF_BUSY semantics, that could produce live-only damage. The brief notes the dispatchBatch handlers all run with the same `cuda_lock` — but the audio thread's `runSynthesisKernel` does NOT acquire `cuda_lock`, so cross-thread interference is possible.

**Verification**: with the live engine, query `getActivePreset()` (must be the working copy), `getLibraryPresets()` (must have both), and instrument `loadParameterToPianoid("dev_dec_open", ...)` to log the slot index it writes into.

### Suspect S2 (MEDIUM, with caveats): dev-midi-p3 / dev-midi-play / dev-midi-p2 — MIDI listener thread

Commits: `2406059` (p3), `3f572db` (midi-play), `e184ba2` (p2), plus `pianoidMidiListener.py` 7-line change.

**What changed**:
- `pianoidMidiListener.py` modified (broadcast gate, listener fixes).
- MIDI listener now runtime-controllable (start/stop endpoints).
- `/play` and WS `play` listener short-circuits REMOVED — `pianoid.listen=True` no longer drops UI play.
- New `_midi_broadcast_enabled` slot + `emit_midi_note_event` per-call filter.

**Could it explain live-only damping failure?** Possibly — the MIDI listener thread now runs concurrently with the LIVE audio thread and the test path. The brief explicitly notes the regression test for dev-midi-play added 5 cases. If the listener thread is broadcasting / echoing inbound MIDI events through `schedule_event`, and the user's hardware MIDI port is open (or self-loops), the listener could re-trigger NOTE_ON for held pitches and mask damping. Note: the user's repro uses REST `/play`, but if the listener thread reads its own queue or echoes events, the live path can be contaminated.

This is the **strongest concurrency-mediated** suspect for a live-only effect.

**Verification**: with the live engine, query `pianoid.listener_running` and the listener thread's port state during the trichotomy repro. If `listener_running == False` and no MIDI hardware is bound, this suspect drops.

## Note — concurrency invariant audit

The audio thread runs `runSynthesisKernel` via `OnlinePlaybackEngine::run()` — without acquiring `Pianoid.cuda_lock`. The middleware acquires `cuda_lock` for ALL preset switches, parameter edits, and the offline `runOfflinePlayback`. If a middleware path re-enters `loadParameterToPianoid` or `cudaMemcpy` to `dev_physical_parameters` / `dev_dec_open` *outside* `cuda_lock` while the audio thread is reading those buffers, the LIVE path will see torn writes; the OFFLINE path (which holds `cuda_lock` around the entire run) cannot. The dev-bfe2 `c2d7c21` introduced `waitForParameterUpdate()` and an in-flight async-upload model — worth checking whether the `OnlinePlaybackEngine::run()` thread is now exposed to a race that the OFFLINE path is shielded from by holding `cuda_lock` across `runOfflinePlayback`.
