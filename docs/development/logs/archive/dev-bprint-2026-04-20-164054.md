### Step 1: Decision (P1/P2) — 16:45
- **P1 Authority:** stdout/stderr are the _Python process's_ output streams; there is no state-ownership rewrite here. We're replacing a direct write to stdout with a call to the module's `logger`. `logger` is already the established observability surface in this file. No ownership change.
- **P2 Concern:** `parse_range` is parameter-range parsing. Emitting a debug trace of "what parameter was requested" is at best diagnostic — not part of the parse concern. This should be a `logger.debug(...)` call (disabled by default) or removed entirely. It's not load-bearing — absence of this line changes nothing about behavior.
- **Scope decision — minimal:**
  1. Primary: replace line 475 `print` with `logger.debug(...)` to match the existing `logger.error` pattern. A `logger.debug` call respects root-logger configuration; if no DEBUG handler is attached, it does nothing silently — crucial since `logger` is not configured at startup in `backendServer.py`, so no output occurs at all, which is the right behavior for this stray trace.
  2. Secondary: add a global `@app.errorhandler(Exception)` that returns JSON with CORS headers (via `after_request` or by letting Flask-CORS attach them on the JSON response). This makes any future 500 readable across origins.
  3. Broader: Do NOT sweep all 87 prints — explicitly out of scope per orchestrator ("open a follow-up doc if scope explodes"). Will add a WIP entry pointing at the backlog.
- **Why `logger.debug` and not guarded `try/except OSError` around print:** a guarded print is a workaround (S5 violation — we know the print is wrong, and swallowing OSError hides other future I/O errors). The logger is the forward direction per the planned migration, so moving this one call there aligns with the plan and does not create technical debt.
