## Step 1 — docs-first (Documentation-First Rule)

Consulted `docs/guides/STARTUP_TROUBLESHOOTING.md` → "Symptom: nvidia-smi reports 'NVML not found'". Confirms the authoritative facts:
- nvml.dll + nvidia-smi.exe are **driver components**, version-locked to the installed display driver; live in System32 + DriverStore.
- "NVML not found" = nvml.dll missing from a loadable location **OR the wrong version for the driver** (← the mismatch case this change adds precision for).
- Fix = reinstall/repair the **DISPLAY DRIVER** (clean install / DDU + reinstall), NOT the CUDA toolkit (toolkit doesn't ship nvml.dll); reboot; re-run.
- check-cuda.ps1 already detects the broken-but-present state (thrown cupy query OR NVML error from nvidia-smi).

The mismatch-specific guidance (reboot FIRST — a driver update without reboot leaves nvlddmkm and user-mode nvml.dll version-mismatched until reboot; DDU clean reinstall of the *matching* driver if reboot doesn't fix it; check for a stray older nvml.dll shadowing System32 on PATH) is consistent with the doc and adds the precise recipe the doc's generic "NVML not found" branch lacks.

## Changes (sections changed) — diagnose-cuda.ps1

**Helpers (new):**
- `Get-NvDriverShort($Version)` — normalises any NVIDIA version string to the `XXX.YY` driver form by stripping non-digits and reading the **last 5 digits** (`...56094` → `560.94`). This is the correct rule: nvml.dll FileVersion (`8.17.15.6094`) and WMI DriverVersion (`32.0.15.6094`) have DIFFERENT lead parts but the SAME last 5 digits = the driver number. (The pre-existing inline regex in section 1, `(\d{3})(\d{2})$`, did NOT actually match these strings — it printed no driver annotation. Now section 1 uses the helper and DOES annotate `~560.94`.)
- Script-scoped capture vars: `$script:NvmlSys32Short/Raw`, `$script:DriverShort/Raw`, `$script:NvmlStoreShorts`, `$script:SmiNvmlError`.

**Section 1 (GPU/WMI):** captures the first NVIDIA GPU's driver version (raw + normalised short) for the section-8 comparison; now prints the `(NVIDIA driver ~XXX.YY)` annotation.

**Section 3 (NVML):**
- Captures System32 nvml.dll FileVersion (raw + short) and DriverStore copies' shorts.
- NEW 3c VERSION COMPARISON: compares System32 nvml.dll short vs driver short → emits a `FAIL` "VERSION MISMATCH: System32 nvml.dll is vX … driver is vY → present but CANNOT LOAD" (with the exact numbers) when they differ, an `OK` "Version match" when equal. Fallback: if the driver short is unavailable, compares System32 vs the DriverStore copy.
- NEW stray-nvml-on-PATH probe: reports any nvml.dll on a **non-System32** PATH directory that could shadow System32 (System32 itself excluded + dirs deduped — this avoided a false-positive that my first live run caught).

**Section 4 (nvidia-smi):** broadened the NVML-error signature to also match "driver/library version mismatch" / "version mismatch"; sets `$script:SmiNvmlError`; when nvml.dll exists in System32 (section 3) AND smi hits an NVML error, prints the present-but-unloadable line with both version numbers.

**Section 8 (DIAGNOSIS):**
- NEW booleans `$nvmlMismatchFlag` (a section-3 VERSION MISMATCH finding) and `$nvmlMismatch` (= that flag OR `$script:SmiNvmlError` while nvml.dll is present in System32).
- NEW `elseif ($nvmlMismatch)` verdict branch inserted **after `$driverFail` and BEFORE the NVML-missing and cupy-blame branches** → this is the precedence fix: a present-but-mismatched nvml.dll is now diagnosed correctly even though nvml.dll exists in System32 (previously `$cupyFail` won and mis-blamed cupy). The branch prints the exact versions + the recipe: (1) REBOOT FIRST; (2) if still broken, DDU clean-reinstall the matching DISPLAY driver (not the CUDA toolkit); (3) remove any stray PATH nvml.dll (conditional on the section-3 finding); (4) reboot + re-run.
- Refined the NVML-missing branch wording to say "MISSING" specifically (mismatch is now handled above).
