### Step A.4: Code analysis — 20:15
The bug must be in the frontend, since all REST-level matrix posts work correctly.
Hypothesis paths examined:

1. **`useMatrixHistory.applyChange` (line 160-165)**: calls `calcChange` TWICE with same input.
   Result: redundant work but deterministic (each call deep-copies). NOT the bug source.

2. **Strings-axis send-path (`useSoundChannels.js:304-309`)**: posts to `/set_parameter/feedback/output` instead of `/set_parameter/string_sound_channel/all`.
   Backend supports both but frontend never uses string_sound_channel. The feedback path uses
   `sm.update_deck(matrix='feedback', pitches=output_pitches, ...)` which validates row length
   (StringMap.py:331). If frontend sends a row of wrong length, backend raises ValueError mid-loop;
   the route swallows the exception (backendServer.py:982) but the deck has been PARTIALLY updated
   and `send_deck_params_to_CUDA()` is never called. Subsequent updates may upload an inconsistent
   deck. Plausible silence vector.

3. **`pitchesVector` in calcChange (useMatrixHistory.js:126-132)**: writes `pitches[change.mode]`
   across all rows. If `change.mode >= row.length`, the row gets extended with `undefined` entries
   and the malformed value at index N. When that matrix is sent to backend, the row length no
   longer matches num_modes → ValueError → silent backend fail.

4. **`modesVector` in calcChange (useMatrixHistory.js:121-126)**: `newMatrix[change.pitch].fill(value)`.
   `Array.prototype.fill()` mutates in place. Since the surrounding code does
   `JSON.parse(JSON.stringify(stageMatrix))` upstream, mutation is on a deep copy. NOT the bug.

5. **MatrixTools inputValue auto-init**: when user clicks "Value" op,
   `newInputValue = value || 0` (MatrixTools.jsx:68). If `value` is null/undefined, inputValue=0,
   so any row/column write would zero the matrix region. This silences sound legitimately. Plausible.

**Most likely culprit**: combination of #2 + #5 in strings-axis mode. User is in strings mode,
clicks "Column"/"Row"+"Value" with a freshly-opened panel where `value` is null → inputValue=0 →
posts a denormalized matrix with a zeroed row/column to `/set_parameter/feedback/output` →
backend writes that row/column to the deck → `send_deck_params_to_CUDA` then uploads → those
output channels' feedback coefficients drop to zero permanently → audio silent. Frontend reload
re-fetches the (corrupted) backend matrix and sees all-zero, so it APPEARS frontend-only restored
because reload triggers a fresh `loadPreset` REST call which resets the backend RAM state from disk.

**Why "never restored" until npm reload**: backend RAM matrix is corrupted; no UI affordance restores
it without a full preset reload. The Undo button only undoes the FRONTEND history, but the
debounced send pushes the post-undo matrix to backend, so undo doesn't restore (and the backend's
already-corrupt state may be re-overwritten with another corrupt matrix). Actual fix likely needs
either (a) inputValue default different from 0, or (b) shape validation before POST, or (c) the
strings-axis path should be `/set_parameter/string_sound_channel/output` instead of feedback.

#### Aggregation choice + reasoning

**Lossless transpose.** The strings-axis canvas is rows = output channels (N=4), cols = piano pitches (84). Storage `string_coefficients[pitchID]` is a per-pitch length-N vector of per-channel gains. The aggregator GET maps `coeffs[c]` (for each pitch p) into row c, column p. The POST does the inverse. No element is summed, averaged, or dropped — every (channel, pitch) cell round-trips exactly. This was the only choice that made physical sense given the data model (`string_coefficients[p][c]` IS the gain for pitch p mapping into channel c) and the only one that makes round-trip idempotent without information loss.
