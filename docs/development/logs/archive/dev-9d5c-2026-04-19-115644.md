## Regression — Bug 2 fix caused infinite feedback loop — 12:22

User reported: top half of stabilization chart constantly redrawing after Bug 2 fix.

### Root cause (confirmed live)

ECharts emits `brushselected` events as a side effect of the `brushVisual` pipeline, which runs on every `dispatchAction({type:"brush",...})` AND every `dispatchAction({type:"takeGlobalCursor",...})`. Ref: `echarts/lib/component/brush/visualEncoding.js:189` calls `dispatchAction(api, throttleType, throttleDelay, brushSelected, payload)` which queues a throttled `brushSelect` -> `brushselected` event.

My first-pass fix introduced TWO always-firing dispatches per brush action:
1. `handleBrushSelected` finally -> `dispatchAction({type:"brush", areas:[]})` -> queues `brushselected`
2. `handleBrushSelected` finally -> `setBrushGeneration(g+1)` -> lifecycle effect runs -> dispatches `brush areas:[]` AND `takeGlobalCursor` -> both queue `brushselected`

On 300ms debounce: ~300ms later the throttled `brushselected` fires with empty `areas`. My handler's inner `if (!areas || areas.length === 0) return;` was INSIDE the try -> finally ran anyway -> re-dispatched brush + re-bumped generation -> loop. Measured before fix: **7 brushselected fires in 2s idle** (confirming the 300ms-period runaway).

### Option chosen — C (echo guard) + revert defensive clear in lifecycle effect

Options analyzed:
- **Option A alone (no generation bump in finally):** does NOT break the loop — the `dispatchAction({type:"brush", areas:[]})` in the finally ALSO triggers `brushVisual` -> queues `brushselected` echo. Even without the generation bump, the finally still re-runs on every echo.
- **Option B (dual counters):** requires discriminator state. More moving parts.
- **Option C (echo detection):** early-return when `brushselected` carries empty `areas` BEFORE entering the try/finally. Real user brush events always carry non-empty areas; empty-areas events are either ECharts echoes or legitimate no-op clears. Either way, there is nothing to act on. Simplest invariant.

Combined Option C with reverting the lifecycle effect's proactive `brush areas:[]` dispatch (added in the first-pass fix). Rationale: the finally block in `handleBrushSelected` is a single, sufficient owner for the brush clear; having a second clear in the lifecycle effect created a second feedback path. Single owner for clear + echo guard at handler entry = loop-free invariant.

### Changes

`PianoidTunner/src/components/StabilizationDiagram.jsx`:

1. **Echo guard at `handleBrushSelected` entry** (lines 1242-1249): moved the empty-areas check OUT of the try, BEFORE any side-effect. Early return terminates the cycle without touching `brushGeneration` or dispatching anything.
2. **Lifecycle effect cleaned up** (lines 976-1008): removed the `dispatchAction({type:"brush", areas:[]})` that was added in the first-pass fix. The effect is now purely responsible for cursor re-arm (takeGlobalCursor). Single-owner rule: brush visual clear is owned solely by `handleBrushSelected` finally.
3. Updated comments in both the lifecycle effect and the `handleBrushSelected` finally to document the single-owner invariant.

## Third-pass fix — orphaned cover from `_creatingCover` guard — 12:32

User reported: "Rectangle brush persists more than before." Worse than first pass. My scripted tests kept passing but real-user interactions exposed a new path.

### DOM-level instrumentation

Installed a 60 Hz sampler on the zrender display list AND on `BrushController._covers / _creatingCover / _dragging` state. Reproduced the bug by simulating a real mouse drag with a 400ms mid-drag pause (longer than the 300ms debounce). Captured:

```
Before fix (mid-drag-pause repro):
- t=256230: Cover rendered (drag start). visible=1, ctrlCovers=1, creating=true, dragging=true
- t=256792: our dispatchAction({type:"brush", areas:[]}) (handler running during the pause)
- t=256815: first brushselected fires (areasLen:1)
- t=256895: lifecycle effect dispatches takeGlobalCursor
- User resumes drag, then mouseup
- t=257511: second brushselected fires (areasLen:0 — echo guard returns)
- FINAL state: visible=1, ctrlCovers=0, groupChildren=1
        ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
        ORPHAN: cover rendered but not in _covers array
```

### Exact root cause (precise this time)

ECharts `BrushController.updateCovers(coverConfigList)` at BrushController.js:189-225:

```js
var oldCovers = this._covers;
var newCovers = this._covers = [];      // _covers emptied IMMEDIATELY
var creatingCover = this._creatingCover;
new DataDiffer(oldCovers, coverConfigList, ...).add(addOrUpdate).update(addOrUpdate).remove(remove).execute();

function remove(oldIndex) {
  if (oldCovers[oldIndex] !== creatingCover) {   // <-- GUARD
    controller.group.remove(oldCovers[oldIndex]);
  }
}
```

When our `dispatchAction({type:"brush", areas:[]})` fires WHILE the user is still mid-drag (they paused >300ms debounce), `_creatingCover` is still the live cover. DataDiffer calls `remove(0)`, but the guard at BrushController.js:221 skips `group.remove` because the cover IS `_creatingCover`. Result:
- `_covers = []` (emptied at line 198)
- Zrender group still contains the cover element — ORPHANED from `_covers` tracking
- User completes drag, mouseup, `_creatingCover = null`, but no subsequent dispatch ever reaches `group.remove(orphan)`:
  - `handleDragEnd` -> `trigger(controller, eventParams)` -> `_onBrush` -> dispatches `brush` with `areas: [...]` derived from `controller._covers` (line 315 of BrushController.js) which is `[]` -> `brushModel.setAreas([])` is a no-op, and the dispatch has `$from: modelId` so `BrushView._updateController` (line 81) SKIPS updateCovers entirely (anti-echo guard).
  - Our second `brushselected` fires with empty areas; my previous echo guard returned without dispatching, leaving the orphan.

My first-pass fix had a second dispatch path (lifecycle effect's proactive clear) that happened to run after the drag ended and caught this orphan. My second-pass fix removed that redundancy, exposing the race.

### Option D (chosen) — direct group reconciliation

Added a `forceRemoveOrphanedCovers()` callback that walks `inst._componentsViews -> brush view -> _brushController`, builds a `Set(_covers)`, iterates `group.children()`, and directly calls `group.remove(child)` for any child not tracked in `_covers`, then `zr.refresh()`. This bypasses DataDiffer's `_creatingCover` guard.

Call sites:
1. **Echo path (top of `handleBrushSelected`):** when brushselected fires with empty areas (post-drag echo), call `forceRemoveOrphanedCovers()`. This is the moment we know the drag has ended (areas already cleared), the perfect time to sweep the orphan.
2. **Finally block:** after the clear dispatch. Handles the case when `_creatingCover` is already null during a fast drag (idempotent no-op).
3. **Lifecycle effect:** at the end, after takeGlobalCursor. Final safety net — runs after React commits, when drag state has typically settled.

Feedback-loop safety unchanged: echo guard still returns without bumping `brushGeneration` or re-dispatching, so no infinite loop.
