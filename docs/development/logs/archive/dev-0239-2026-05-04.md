## Root cause — user-side flow mismatch

The Modal Adapter UI exposes **two distinct project-creation flows** that take **different file shapes**:

### Flow A — "Import Project" button (UI: FileUploadIcon)

- Calls `POST /modal/projects/import` (multipart upload)
- Handler `routes.py:288-316` → `modal_adapter.import_project(zip_path)` (`modal_adapter.py:708-773`)
- **Required input:** a `.pianoid-project` zip archive previously produced by the **"Export Project"** button, which contains a top-level `manifest.json` (with `format_version >= 1`) and `project.json`
- **What it rejects:** any zip without `manifest.json` → HTTP 400 `{"error": "Invalid archive: missing manifest.json"}` (line 727)
- File-input accept attribute: `.pianoid-project,.zip` (`ModalAdapter.jsx:705`) — this is the trap: the picker accepts any `.zip`, but the handler only accepts the export format

### Flow B — "Create Project" panel (per `MODAL_ADAPTER_GUIDE.md` "Project Management → Create new")

- Calls `POST /modal/projects/create` with `{name, source: <folder_path>}`
- Handler `routes.py:147-160` → `modal_adapter.create_project()` → `_import_measurements()` → `_discover_roomresponse_scenarios()` (`modal_adapter.py:1244+`)
- **Required input:** filesystem path to the **un-zipped** RoomResponse parent folder containing `<Whatever>-Scenario<N>-<Whatever>/averaged_responses/average_chN.npy`
- Folder regex `r'Scenario(\d+)'` (`modal_adapter.py:1241`) matches `PlyWood-Scenario0-Take1` → idx 0. The user's data is structurally correct for this flow.

The user used Flow A (Import Project) with a Flow B input (a regular zip of a RoomResponse measurement folder). The handler hits its first format validation and returns HTTP 400.

## Optional follow-up — UX hardening (separate /dev session, NOT for this one)

The current "Import Project" button silently accepts any `.zip` from the OS file picker but rejects anything that isn't a `.pianoid-project` archive. This trap will hit other users. Possible improvements (any combination):

1. **Frontend:** change `accept=".pianoid-project,.zip"` to `accept=".pianoid-project"` only, OR add a hint text under the button that says "Use 'Create Project' for raw measurement folders"
2. **Backend:** when `manifest.json` is missing, sniff for top-level `*-Scenario<N>-*/averaged_responses/` directories and return a more helpful error like `"This looks like a RoomResponse measurement folder, not a .pianoid-project archive. Use POST /modal/projects/create with the unzipped folder path instead."`
3. **Backend (most useful):** add a `POST /modal/projects/create_from_zip` endpoint that auto-extracts the user's measurement-folder zip to a temp dir then invokes `create_project(source=temp_dir)`. Handles the user's exact mental model "I have a zip, I want a project from it."

These would have prevented this user's confusion. Filing as a follow-up; **not** changing code in this session per the task constraint (don't gold-plate).

## Phase 5 — Canonical-pipeline averaging (RoomResponseRecorder pipeline)

User feedback after Phase 3 ("expecting 30 scenarios") and Phase 4 ("the averaging has to be done properly"). Implementation:

### What was added

**New module** `PianoidCore/pianoid_middleware/modal_adapter/scenario_averager.py` (~370 LOC). Re-implements the canonical 6-step `RoomResponseRecorder` averaging pipeline (validate → align → normalize → average → truncate-with-fadeout) by orchestrating the pure-numpy helpers from the sibling RoomResponse repo: `signal_processor.SignalProcessor` (extract_cycles, align_cycles_by_onset, apply_alignment_to_channel, normalize_by_calibration, average_cycles, truncate_with_fadeout) and `calibration_validator_v2.CalibrationValidatorV2`. Operates on a scenario folder's `raw_recordings/` + `metadata/<scenario>_metadata.json`.

Public API: `ensure_averaged_responses(scenario_dir, *, force=False) -> AveragingResult`, `ensure_averaged_responses_for_parent(parent_dir) -> Dict[str, AveragingResult]`, `summarise_results(results) -> dict`. Idempotent — pre-existing `averaged_responses/*.npy` are skipped (never overwritten unless `force=True`).

**Wired into** `ModalAdapter.create_project_from_zip` via new helper `_auto_average_scenarios(root: Path)` — runs after extraction, before measurement-source detection. Response now includes `averaging_summary` field with counts.

**Frontend** (`PianoidTunner/src/modules/ModalAdapter.jsx`): success alert now reports `Scenarios imported: N` plus the averaging breakdown (`computed: N`, `skipped_existing: N`, `errors: N` if any).

**Docs** (`docs/guides/MODAL_ADAPTER_GUIDE.md`): new "Auto-averaging missing averaged_responses/" subsection documents the 6-step pipeline, idempotency, and response shape.
