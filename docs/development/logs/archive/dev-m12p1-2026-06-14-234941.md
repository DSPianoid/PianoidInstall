**P1 (Authority) — who owns each piece of state (sole writer):**
- Inbound delivery queue: the `DeliveryQueue` owns the on-disk queue dir (enqueue/ack/replay) — sole writer. The adapter calls it; nothing else writes the queue files.
- Capture store: the `CaptureStore` owns its append-only log file — sole writer (append + read only; no in-place rewrite).
- I/O bus: the `IoBus` owns subscriber fan-out; it holds no durable state (pure in-memory broker). Producers publish, the bus fans out; it never mutates message content.
- Adapter registry: the `Supervisor` owns the registry map (register/start/stop); adapters never register themselves elsewhere.
- All NEW files in an isolated subtree (`tools/supervisor/`); no existing Pianoid/plugin state is written (Phase 1 is additive). The live plugin's `access.json`/`.env`/`inbox/` are READ-ONLY references for shape — the supervisor uses its OWN state dir for its own queue/capture so it never contends with the live bot.

**P2 (Concern) — one job per module:**
- `IoBus` — marshal/fan-out messages (no interpretation).
- `ChannelAdapter` (interface) — the contract only; no impl.
- `DeliveryQueue` — durable, ack'd, replayable inbound persistence (the inbox-queue patch's job, first-class).
- `CaptureStore` — durable append-only replayable event store (the stream-json capture).
- `TelegramAdapter` — Telegram transport (grammY) behind the contract; composes DeliveryQueue + voice helpers; does NOT own the bus or capture.
- `VoiceCodec` (STT/TTS) — voice↔text only, behind the adapter (re-homes transcribe_voice.py / tts_voice.py as helper-process calls).
- `Supervisor` — lifecycle/orchestration of bus+adapters+capture (no transport logic, no Claude subprocess yet — that's Phase 2).
- `Transport` abstraction — lets the Telegram adapter run on a real grammY transport OR a deterministic loopback/mock transport for tests (the safety lever: tests NEVER open a live poller on the production token).
