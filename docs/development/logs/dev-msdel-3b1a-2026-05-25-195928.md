Even the pathologically-huge 8.8 GB / 2526-file measurement deletes in ~1 s on this disk. **The 5000 ms budget is not failing because of disk speed alone.**

#### Why the user STILL hits 5000 ms — second factor

`modal_adapter_server.py:268` runs `app.run(threaded=False)` (deliberate: CuPy GPU requires the main thread). EVERY request to port 5001 is serialised. The frontend has several concurrent pollers + long-running operations on the same port:

- `useImportSession` polls `/modal/import_operations/<op_id>/status` every 500 ms
- `useServerLifecycle` poll (interval)
- `useModalAdapter` pipeline poll on `/modal/status`
- FRF runs (~22 s per `useModalMassRun` round 2 commit notes)
- Modal-mass orchestrator runs (~several seconds)
- ESPRIT runs (10s+)

If the user clicks Delete while ANY of these is in flight, the DELETE request queues behind it. A 5 s axios timeout is unsurvivable if the user happens to delete during an FRF run. The sibling `renameMeasurement` already uses 30000 ms with the comment "rollback path can touch many files" — same disk-cost profile, same serialisation risk, but generously budgeted.

#### Root cause classification

**(a) frontend-timeout-too-tight.** 5000 ms was set in `dev-maimport` round 14 (commit `2c65ad4`, "Measurements browse + delete management dialog"). Backend handler itself is correct (refuses 409 on referenced measurements per N6, `shutil.rmtree` on `path`, no cascade/background work). Other delete-class operations in the same hook use 30000 ms (`renameMeasurement`); sibling `useProjectCRUD.deleteProject` has no explicit timeout (axios default = no timeout). The 5 s on measurement delete is anomalously short.

#### Fix

Bump `deleteMeasurement` timeout from `5000` → `60000` ms (60 s):
- Comfortably covers worst-case `rmtree` + queue-behind-other-request scenarios
- Matches `useProjectCRUD.fetchQCCurves` (60_000) — the closest sibling that has a non-default explicit timeout
- The existing `busy` state in `DeleteMeasurementConfirmDialog` already shows "Deleting…" so UX is fine
- Add regression Jest test asserting the new timeout value
- Backend handler + Python tests untouched (they're correct)
