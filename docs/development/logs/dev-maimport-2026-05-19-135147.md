**Issue 1 — Dialog closes immediately** (root cause found, fix is small):
- Frontend bundle freshness: PROVED loaded. Grep'd `http://127.0.0.1:3000/static/js/bundle.js` for round-7 sentinels: `ResultPanel` = 5 hits, `Click OK to dismiss` = 1 hit, round-5 sanity sentinel present. The new code IS in the bundle.
- Root cause: parent-side lifecycle bug. `ModalAdapter.jsx:1151` conditionally renders `<CollectionSubpanel>` only while `activeSection === "collect"`. The dialog state (`newProjectDialogOpen`) is OWNED by CollectionSubpanel. On round-7 success path, `handleSubmit` calls `setResult(...)` then `onSwitchToProject()` — which flips `activeSection` to "setup" → CollectionSubpanel UNMOUNTS → dialog destroyed mid-render with its result panel never displayed. User sees "dialog flashed and closed".
- Fix: defer `onSwitchToProject()` from `handleSubmit` to `handleOk` (so result panel stays mounted until user dismisses it; switch-to-Setup happens on OK click).

**Issue 2 — Project created + averaging displayed**: round-7 fix WORKING. No further action.

**Issue 3 — Band config locked**: downstream of Issue 4. Locked when 0 modes exist. If Issue 4 unblocks ESPRIT → Issue 3 resolves automatically. Open question: should band config be editable prospectively before any ESPRIT run? Need user direction.

**Issue 4 — ESPRIT runs too fast, 0 modes** (root cause found):
- ESPRIT input data is REAL (not zeros, not NaN). Inspected user's `D:\modal_measurements\PlyWood\scenarios\PlyWood-Scenario1-LG\averaged_responses\average_ch*.npy`: ch0 (calibration) abs_max=0.95 std=0.011; ch1-7 (would-be response) abs_max=0.08-0.13 std=0.008-0.012. 28800 samples = 600ms at 48kHz. Real piano signal.
- Root cause: user's `setup/mapping_config.json` has `channel_roles: {"0": "force"}` ONLY. Channels 1-7 have NO role. So `Mapping.response_channels` is EMPTY. `ModalAdapter._run_esprit_sync:3447` then computes `response_channels = mapping.response_channels` → empty list. ESPRIT "succeeds" in the sense that the loop finishes, but it has no response channels to extract modes from → 0 per-scenario modes. **HTTP 200 state=done is misleading** — it should error out with "No response channels configured in mapping".
- Compounding: user's mapping_config also has `layout_type: "grid"` with `grid_shape: [12, 15]` = 180 cells but only 3 scenarios actually exist. Grossly mismatched grid.
- The mapping_config was NOT generated by my round-1 import (file says `created_by: "ModalAdapter Phase 1"`, not dev-maimport). The user's PlyWood Measurement was created via the legacy "+ New Measurement" empty-create then scenarios added separately. So my import flow didn't touch this Measurement's mapping. **But round-1 import_folder DOES default to `default_mapping_config()` which is empty `channel_roles: {}` — same downstream problem for any new import.**
- Fix has two parts:
  (a) `_run_esprit_sync` should emit a clear error when `response_channels` is empty after mapping derivation (instead of silently returning 0 modes). This is a defensive fix that surfaces the real problem in any future case.
  (b) `import_folder_as_measurement` should generate a sensible default `channel_roles` for the imported Measurement when the source's `session_metadata.json::recorder_config.multichannel_config` carries `calibration_channel` + `response_channels` keys (which the user's source DOES carry — same data round-1 already reads for audio_config).

**Issue 5 — Tracking blank** (small UI fix):
