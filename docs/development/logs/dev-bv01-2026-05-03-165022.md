### Deviations from the original plan:
1. The plan's clamp at "REST/WS handler then call schedule_event with
   the clamped value" was refactored mid-implementation: I added an
   `apply_fix_velocity` parameter to `schedule_event` itself so the
   gating is centralized inside the helper. This avoids leaking the
   default-clamp behaviour to non-MIDI callers via the unified MIDI
   listener path (which calls schedule_event directly with default
   `apply_fix_velocity=True`). REST/WS handlers now pass
   `apply_fix_velocity=is_midi_source` and the helper does the clamp
   gate. `perform_midi_command` defaults to `apply_fix_velocity=False`
   so legacy callers (NoteTunner calibration, set_sustain) preserve
   their explicit velocity. This is a strictly cleaner design than
   the plan's parallel-clamp approach — same observable behaviour,
   single gate point.

2. NoteTunner's `fixed_level` rename was kept smaller per plan (D3) —
   single underscore prefix `_calibration_level`, no algorithm change.
