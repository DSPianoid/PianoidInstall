## Design evolution (2026-06-11, post-first-push) — MULTI-PROJECT + in-place (ii) separation

After the first push (commit `6335869`), two requirements reshaped the P1 separation. **This is a DESIGN checkpoint** — the ~45 generic-skill edits are designed + apply-ready but **NOT yet executed**; what is *done* is the design + the groundwork (`PROJECT_CONFIG.md`, the manifest, the script alias). Full detail: **`docs/proposals/generic-project-separation-plan-2026-06-11.md`** (the dedicated plan + Addenda 1–3), with `docs/PROJECT_CONFIG.md` (the complete 17-anchor project-config SSOT) and `docs/development/GENERIC_CORE_MANIFEST.md` (the per-artifact lift-list).

1. **MULTI-PROJECT is now a first-class requirement** (user: *"one system can manage several projects; one orchestrator may run two SIMULTANEOUSLY"*). A generic section duplicated inside each project's `CLAUDE.md` would be cross-project drift one level up — so the **generic core must be a SHARED unit (one copy across projects), not per-project.** Validated against the official Claude Code docs (memory/settings/sub-agents/agent-sdk):
   - **User-level `~/.claude/` is the shared layer** — `~/.claude/CLAUDE.md` + `~/.claude/commands/` + `~/.claude/rules/` load for *every* project regardless of cwd. Project-level (`<repo>/.claude/`) loads per-project by walking up from cwd. User + project are **additive** (no hard precedence → "state precedence explicitly in the more specific file").
   - **THE CRUX (simultaneous two-project):** per-subagent `cwd` is an **UNSHIPPED** Claude Code capability (open issues #31940/#12748) — sub-agents inherit the parent's single cwd. So correct per-project config resolution under concurrency **cannot be implicit (cwd-derived); it requires an EXPLICIT `PROJECT_ROOT` binding passed in every dispatch brief** — a new hard orchestrator rule (on the order of the `bypassPermissions` rule).
   - **Counterintuitive precedence (verified):** when a skill name exists at both levels, **personal (user) OVERRIDES project** — so the eventual hoist must *delete* the project copy of a hoisted skill (else it is silently shadowed).
   - **Three-layer design:** USER `~/.claude/` = shared project-agnostic generic core · PROJECT `<repo>/` = its own `PROJECT_CONFIG.md` + project rules + project-specific skills · DISPATCH = explicit per-task `PROJECT_ROOT` binding.

2. **P1 is executed IN-PLACE via (ii) stage-in-repo-first** (DP-1 = (ii), over (i) immediate hoist), with the **mark-in-place** mechanism. Verified: Claude Code does **not** load skills/commands from a separate top-level dir (only `.claude/commands` + `~/.claude/commands`), and `.claude/commands/` symlinks are undocumented + Windows-fragile (empirically `ln -s` here makes a copy, and native symlinks need Dev-Mode). So the generic skills **stay physically in `.claude/commands/`, tagged `tier: generic` and made genuinely project-agnostic** (every Pianoid fact → a `$PROJECT_ROOT/docs/PROJECT_CONFIG.md#anchor` reference), proven by a **grep gate** (zero `pianoid`/`PianoidCore`/port-quad/`build_pianoid`/`note_playback` tokens in a generic skill = machine-checkable liftability). The generic `CLAUDE.md` rules split into a new `.claude/CLAUDE.generic.md` (`@`-imported by `.claude/CLAUDE.md`). **The physical hoist to `~/.claude/` is a later, separately-approved step** (the agnostic-ization is the real value + is location-independent; the move is then trivial).

> **`/multitask` folded into the orchestrator (2026-06-12):** `/multitask` is **not** a separate module — the orchestrator is **multi-task by nature**, so a standalone multi-task skill is redundant. Its capabilities (classify + schedule multiple tasks into concurrent **waves**, per-agent git **worktree isolation** for same-repo parallelism, and the post-parallel **merge sweep**) are folded into **Module 1 (Orchestrator)** as coordination responsibilities (reinforcing AP1/AP6/CP5/CP6). The Topic-4 module numbering therefore skips `/multitask`: after M3 `/fn`, the next module is `/review` (M4).

> **Drift-lint folded into dev-pipeline tooling (2026-06-12):** the drift-lint is **not** a separate module — it is a **dev-pipeline script** with an **SSOT-enforcement role**, so it is folded into **Module 8 (Dev-pipeline tooling)** rather than standing alone. As the deterministic CI/pre-commit guard that fails the build when a forbidden/duplicated form reappears (stale build command, an inlined fact that belongs in the config, a skill name shadowing a built-in), it is the **can't-recur guarantee** for the SSOT consolidation — reinforcing M8's **AP2** (SSOT, in its *enforcement* mode) and adding **CP2** (it enforces the docs/config single-source) to M8's traces. The Topic-4 module numbering therefore skips a standalone drift-lint: after M8, the next module is the Version Control Manager (M9).

> **FK (version-control flow) added + `/sync` reframed as the Version Control Manager (2026-06-12):** a derivation correction. CP4 was **under-derived** — the flow set realised it only via "FE records" + "this doc's hierarchy," with no dedicated flow, on the reasoning that version-control/traceability was "cross-cutting." That conflated two facets: the **traceability PROPERTY** (logging + attribution) *is* cross-cutting (woven through every flow), but the **version-control PROCESS** (the git lifecycle: branch → commit → merge → reconcile → push → tag → history/recovery) is a recurring **sequence** and therefore — by §0.6's completeness rule (*a principle implying a recurring sequence yields a flow*) — **is a flow: FK** (realises CP4; + CP7 safety on push/merge, CP5 pre-push gate). Correspondingly, **Module 9 is reframed from `/sync` to the Version Control Manager** (#mod-vcs): `/sync` (clean→commit→merge→push) is **one capability**; the manager owns the **full git lifecycle** across the project's repos. FK is propagated to the modules in that lifecycle — M9 (owns FK), M2 `/dev` wrap-up (commit/merge), M1 Orchestrator (the merge-sweep).

> **`/update` dissolved by concern (2026-06-12):** `/update` is **not** a module — it splits along the **codebase ↔ binaries/environment** concern boundary. The **codebase-update** half (pull / reconcile from origin — *what the source is*) goes to the **Version Control Manager (M9)**: M9 already owns the git lifecycle, and pulling updates from origin is a codebase operation. The **rebuild-binaries-and-environment** half (*what is built from that source*) goes to **dev-pipeline tooling (M8)** — its build script is the **deterministic REBUILD function** (rebuild binaries + sync the environment based on a codebase update; the post-pull/post-merge rebuild gate: detect a compiled-file change in the pulled diff → rebuild → smoke-test; traces CP5/CP6/CP4). The **pull → rebuild orchestration** is simply M9 then M8 in sequence (the orchestrator coordinates; the rebuild gate is M8's function triggered by an M9 codebase change). The M10 `/update` placeholder is removed; numbering continues with the Channel adapter (M10). **The governing rule recorded: module boundaries follow CONCERNS (codebase vs binaries/environment), not legacy skill names** — the same principle that dissolved `/multitask` (→ M1) and folded the drift-lint (→ M8).

> **Module 11 reframed as the PROJECT LAYER + the generic/project split sharpened (2026-06-12):** Module 11 is **not** just `PROJECT_CONFIG.md` — it is the **Project Layer**, three pieces: **(a) FACTS** (`PROJECT_CONFIG.md` — the operational SSOT the skills *resolve*), **(b) project NORMATIVE RULES** (project `.claude/CLAUDE.md` — the project-specific principles/methodologies/preferences the skills *observe*: quality standards, UI/design conventions, testing methodology, domain cautions like the high-stakes-inference/axis-semantics category, workflow preferences — which extend/override the generic principles per §0.3), **(c) the init tool** (`devkit init`, GENERIC — interrogates the project and generates BOTH). The **load-bearing design point recorded**: the generic/project split is **NOT data-vs-code — it is generic-METHODOLOGY vs project-(FACTS + project-specific-NORMATIVE-RULES); the generic skills OBSERVE the project's principles, not only its facts.** This sharpened **AP3** (§0.5 — the boundary now states facts *and* rules), added the **project-rules-refine-the-generic-principles precedence** note to **§0.2** and **§0.3**. Tier: the two generated files are **PROJECT**, the init tool is **GENERIC**. *(Superseded by procedure note #13, 2026-06-13: the "Project Layer" was a layer, not a module — it was SPLIT: the **Init tool** is now Module 11; the project config/facts → §0.6 (structure); the project-rules-observance → §0.10(c). The substance above is unchanged, only its home.)*

> **Development-cycle taxonomy added as §0.9 (upstream classification) + MULTIPLE taxonomies recorded (2026-06-13):** a foundational **work-kind classification** placed **upstream of the modules** (§0.9, after the §0.7 matrix / before §0.8) — it is *classification, not a flow*. Six categories on the dimensions *needs-a-proposal? · phased? · changes-code? · output-type*: **CAMPAIGN** (proposal→phased→debug; yes/yes/yes), **FEATURE** (direct from prompt; no/no/yes), **BUGFIX** (HOTFIX folded in — urgency = priority, not a different workflow; no/—/yes), **INVESTIGATION** (read-only; understand how/why; may yield a proposal→Campaign), **AUDIT** (read-only; findings vs a rubric; realised by `/review`/M4), **CHORE** (light maintenance / docs-only). Cross-referenced from **M1** (classifies a task into a category to route it) and **§0.10(a)** (the working-doc lifecycle, parameterized by category).

> **Procedure note #13 — MODULES vs CROSS-CUTTING PROCESSES/RULES vs STRUCTURES/LAYERS (2026-06-13):** the module-by-module topic (§0.8) holds **only true MODULES** — *functional components with clear responsibility borders*. Three element-kinds that are NOT modules were distinguished and re-homed so §0.8 stays clean: **CROSS-CUTTING PROCESSES & RULES → new §0.10** (the **working-documentation lifecycle** [was "M13"], the **proposal lifecycle**, and the **project-rules-observance** [the normative part flagged at the old M11] — they apply *across* the system, not owned by one component); **STRUCTURES / LAYERS → §0.6** (the **project config/facts** is the PROJECT layer, a structure the skills resolve — referenced in §0.6, not a module); and the already-upstream **CLASSIFICATIONS / FLOWS / PRINCIPLES** (§0.9 / §0.6 / §0.2). Consequently **M11 "Project Layer" was SPLIT**: the **Init tool** stays a module (M11, it *generates* the layer); the config/facts → §0.6; the project-rules-observance → §0.10(c). The §0.8 matrix keeps a one-line "→ §0.10 (process)" / "→ §0.6 (structure)" pointer for each re-homed element so nothing is lost. This is re-homing + re-labelling of **approved content**, not a rewrite. *(Same governance discipline as the earlier "capability/concern, not legacy skill name" folds — here the axis is element-KIND: component vs process/rule vs structure.)*

> **Procedure note #14 — the TRANSITION GRAPH comes before the lifecycles (2026-06-13):** the dev-cycle categories (§0.9) are **nodes in a directed graph**, not isolated buckets — embedded as **§0.9.x**. Entry points (direct→Feature/Bugfix/Chore; research→Investigation; quality-concern→Audit COLD; major-effort→proposal→Campaign) + four transition kinds (**SPAWN** a set of children, **ESCALATE** to a heavier category, **NEST** phases inside a Campaign, **DETOUR** to an Investigation then resume) + the full per-category transition table (entry · edges · output doc). Invariants: **every category can TERMINATE** (no forced edges); **chains compose** (Investigation→Audit→{Features}). This is placed **before the §0.10 lifecycles** because the lifecycles thread parent→child along these edges — you need the graph to know which docs relate.

> **Procedure note #15 — the doc hierarchy is LOGICAL (authority/governance), NOT historical (spawn-order) (2026-06-13):** §0.10(0) Part 1. Authority ranking: **Campaign proposal = GOVERNING (top)** > Audit/Investigation outputs (middle) > Feature/Bugfix/Chore operational records (bottom). Three relationship types a doc can hold: **GOVERNS** (senior rules junior — proposal→phases), **TRACKS** (senior lists+status-tracks a self-governed junior — audit→remediation items), **REFERENCES** (citation, no authority). The load-bearing rule: a **higher-authority spawned doc DEMOTES its spawner to a REFERENCE** (an Investigation that spawns a Campaign is demoted; the proposal becomes senior); an investigation/audit that must **GOVERN** is **promoted to a proposal** (governing authority = proposal-tier). Matches PART-0's most-foundational-governs-downstream spine.

> **Procedure note #16 — DURABLE vs EPHEMERAL doc tiers + the NEW Implementation Report (2026-06-13):** §0.10(0) Part 2. Three tiers by durability×audience: **T1 durable deliverables** (human-readable, archived, SURVIVE wrap-up: **Proposal** [1/campaign, never splits unless the campaign splits], the **NEW Implementation Report** [the as-built OUTCOME, authored at wrap-up, distilled from the ephemerals — the proposal's plan↔outcome counterpart], Investigation findings, Audit report); **T2 working planning** (agent-readable, EPHEMERAL, governed by the proposal: phase plans); **T3 operational logs** (agent-readable, EPHEMERAL: session logs, distilled into the T1 report then discarded). The plan↔outcome pairing is what makes T2/T3 safely discardable. **Implementation Report added as a doc category** (matrix pointer row → §0.10(0)).

> **Procedure note #17 — the DURABLE-TRACE FLOOR (CP4 at the line level) (2026-06-13):** §0.10(0) Part 3. Every code-changing item — *including a standalone Feature/Bugfix with no proposal and no report* — leaves a **durable trace to user intention** via the **structured commit record** (intention + change + verification), persisted in git. Chain: **user-intention → work-item → commit → diff → line**, walked backwards by `git blame`. This is **why the ephemeral T3 log is safely discardable** — the commit carries the durable essence. Scales by category: Campaign→Implementation Report + commits; Feature/Bugfix→commit record; Chore→lightest commit; spawned→ + a parent tracking link. Realises CP4 at the finest grain: no code change anywhere without a durable, attributable, intention-bearing trace.

> **Procedure note #18 — the lifecycle synthesis embedded; PART 0 structurally COMPLETE (2026-06-13):** the **capstone** — the per-category lifecycle synthesis replaced the §0.10(a)/(b) placeholders with full bodies, consuming §0.10(0) (doc-model) + §0.9.x (graph). **§0.10(a) working-documentation lifecycle** is now the canonical *entry → during → wrap-up → durable-footprint* table per category (Campaign / Feature·Bugfix / Investigation / Audit / Chore) + the **transition overlay** (SPAWN → parent TRACKS, child REFERENCES; ESCALATE → doc PROMOTED, origin preserved; spawned proposal GOVERNS + demotes spawner) — parameterized by §0.9, guaranteeing T1 durability + the §0.10(0) trace floor. **§0.10(b) proposal lifecycle** is now the four stages — **DRAFT** (in `docs/proposals/`, `/analyse`-grounded) → **REVIEW & APPROVAL** (governed top-down PART-0-first, one question at a time; the standardized approval-procedure doc governs HOW — forthcoming) → **ACTIVE/IMPLEMENTATION** (the approved proposal GOVERNS; never splits unless the Campaign splits) → **ARCHIVE** (`git mv` to `proposals/archive/` + Status line + author the Implementation Report). This **completes the proposal's structural content** (PART 0 foundations → principles → architecture → modules → cross-cutting processes are all embedded; what remains downstream is Parts 1–3 detail + the still-forthcoming standardized approval-procedure doc).

## The composition method these notes encode (skill seed)

The reusable procedure that produced the proposal, distilled from the notes above:

1. **Top-down, governed, topic-by-topic review.** Foundations first (purpose → principles → decision hierarchy → architecture → flows) → classification (taxonomy + transition graph) → modules one at a time → cross-cutting processes. The user approves/comments per topic before the next; one question at a time.
2. **`traces-to:` is the spine.** Every downstream element traces to ≥1 upstream principle; the matrices are derived; upstream-governs-downstream; grep an ID to find all dependents.
3. **Element-kind discipline (note #13).** Keep three kinds apart: MODULES (functional components, clear borders), CROSS-CUTTING PROCESSES/RULES, STRUCTURES/LAYERS. Module boundaries follow **concerns/capabilities, not legacy skill names** (hence the folds: `/multitask`→orchestrator, drift-lint→tooling, `/update`→VCS+tooling, `cli-control`→supervisor).
4. **Transition-graph before lifecycles (note #14).** Define how work-kinds flow into one another before the document lifecycles, because the lifecycles thread parent→child along those edges.
5. **Logical-not-historical doc hierarchy (note #15)** + **durable vs ephemeral tiers (note #16)** + **durable-trace floor (note #17).** Authority is governance, not spawn-order; T1 survives wrap-up, T2/T3 are ephemeral; every code change leaves a durable intention-bearing commit trace.
